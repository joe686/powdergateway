## 一、产品定位

### 1.1 一句话定义

**PowerGateway 是一款企业级接口通用外挂**，具备两大独立引擎：**报文转换引擎**（替代传统 Gateway 的适配层）和**可视化接口开发引擎**（替代低代码接口开发平台），两者可独立使用也可组合使用。

### 1.2 双引擎定位

```
引擎一：报文转换引擎 — 替代 Gateway 的适配层
引擎二：可视化接口开发引擎 — 替代低代码接口开发
```

PG 不是 ESB 的替代品，而是与 ESB **互补**的：
- ESB 负责：报文路由分发、消息队列、服务编排
- PG 负责：接入 ESB 之前的报文标准化 + 独立的增量接口开发

### 1.3 两大核心场景

**场景 A：报文转换（替代 Gateway 适配层）**

```
业务系统A --原始报文--> PowerGateway --标准化报文--> ESB --> 业务系统B
```

- 痛点：业务系统A接入 ESB 需要改造报文格式、字段命名、编码方式
- 传统做法：改业务系统A的代码适配 ESB 规范，开发+测试+上线周期长
- PG 做法：在 A 和 ESB 之间加一层桥梁，可视化配置报文转换规则，A 的代码零改动
- 价值：**不改一行代码，让已有系统快速接入企业集成总线**

**场景 B：快速增量接口开发（替代低代码接口开发）**

```
调用方 --> PowerGateway --动态SQL--> 业务数据库
```

- 痛点：系统上线后总有增量需求（加查询、改字段、导数据），每个都要走完整开发流程
- 传统做法：写代码 -> 测试 -> 部署 -> 注册到注册中心，单个接口至少半天
- PG 做法：可视化配置 SQL 和参数映射 -> 发布 -> 可调用，10 分钟搞定
- 价值：**把增量接口的开发工时从半天压缩到 10 分钟**

### 1.4 关键指标

| 指标 | 数值 |
|------|------|
| 交付单元 | 28（全部完成） |
| 自动化测试用例 | 326（全绿） |
| REST 控制器 | 20 |
| 数据库表 | 15 |
| 报文格式互转组合 | 12 种 |

### 1.5 技术栈

**后端：** Spring Boot 2.7.18 + MyBatis-Plus 3.5.7 + Sa-Token 1.37.0 + Caffeine + Redis + dynamic-datasource + JDK 1.8+

**前端：** Vue 3.4 + Vite 5.2 + Element Plus 2.7 + Pinia + ECharts 5.5

## 二、能力图谱

### 2.1 引擎一：报文转换（对标 Gateway 适配层）

| 能力 | 说明 | 价值 |
|------|------|------|
| 格式转换 | JSON/XML/CSV/FormData 四种格式 12 种互转 | 解决不同系统间报文格式不统一 |
| 字段映射 | 拖拽配置源字段到目标字段映射 | 解决字段命名规范不一致 |
| 字段加工 | Trim/Pad/Substring/Case/TypeCast 策略模式 | 解决数据格式差异（日期格式、编码长度） |
| 渠道模板 | 渠道码绑定识别字段+模板，运行时自动路由 | 同一接口服务多个渠道，按渠道码选择不同转换规则 |
| 端口分发 | 请求加工 -> 转发 -> 应答加工的完整链路 | 作为代理层转发请求，对上下游透明 |
| 编码转换 | GBK/UTF-8 互转 | 解决老旧系统编码问题 |

### 2.2 引擎二：可视化接口开发（对标低代码接口开发）

| 能力 | 说明 | 价值 |
|------|------|------|
| 数据库直连 | MySQL/Oracle/PG，AES-128 加密，动态切换 | 直连业务库，无需额外部署 |
| SQL 配置生成 | QueryBuilder/InsertBuilder/UpdateBuilder/DeleteBuilder | 可视化配置替代写代码 |
| 接口发布 | draft -> published -> disabled 状态机 | 统一管理接口生命周期 |
| 分库分表 | 取模/范围/补查路由策略 | 支持复杂分片场景 |
| 双层缓存 | Caffeine -> Redis -> DB + 分布式锁 | 高频查询场景直接提升性能 |

### 2.3 运维支撑

| 能力 | 说明 |
|------|------|
| 三库分离 | 配置库 / 业务库 / 审计库 职责完全隔离 |
| AOP 异步审计 | @AuditLog / @SysLogRecord / @PerfStat 三套独立切面 |
| 性能监控 | ECharts 可视化 + 定时告警 |
| SQL 审计 | 记录 SQL 文本、执行前快照、操作 IP |

## 三、竞品分析（双引擎分别对标）

### 3.1 引擎一竞品：Gateway / API 适配层类产品

报文转换引擎对标的是**在 ESB 之前做报文适配的 Gateway 和集成工具**。

#### MuleSoft Anypoint

- **定位**：Salesforce 旗下企业级集成平台
- **核心能力**：DataWeave 数据转换（极强）、Flow 编排、API 管理
- **价格**：年订阅数万至数十万美元
- **PG 差异化**：PG 只做转换不做编排，更轻量、更易用、成本极低

- **定位**：Apache 开源集成框架
- **核心能力**：200+ 组件、路由规则 (DSL)、数据转换
- **价格**：免费开源

## 五、优劣势分析

### 5.1 核心优势

1. **双引擎合一** — 报文转换 + 接口开发在同一个平台，竞品要么只做转换要么只做开发
2. **极致轻量** — Spring Boot 单体部署，一个 JAR 包启动
3. **全部可视化** — 报文转换、字段映射、SQL 配置全部 Web 界面完成，零代码
4. **分库分表独有** — 接口开发赛道中几乎没有竞品具备此能力
5. **学习成本极低** — 不需要理解 ESB 概念，会写 SQL 就能用
6. **三库分离架构** — 配置/业务/审计隔离，安全性和可维护性优秀
7. **326 个自动化测试** — TDD 质量保障

### 5.2 核心劣势

1. **缺少服务注册发现** — 无 Nacos/Eureka/Consul 支持（**最紧迫短板**）
2. **无 Mock 服务** — 缺少接口 Mock 能力
3. **接口编排能力弱** — 不支持复杂的多步骤编排和条件分支
4. **缺少批量导入导出** — AUX-3/4/5 仍处于待规划状态
5. **技术栈偏旧** — Spring Boot 2.7 + JDK 8（已有升级计划，非紧迫）

### 5.3 定位修正记录

| 版本 | 修正内容 | 原因 |
|------|---------|------|
| v1.0 | 初始定位为"低代码可视化接口开发平台" | 基于功能描述 |
| v1.1 | 修正为"企业级内网使用"，端口分发降级 | 用户反馈定位为企业内网 |
| v2.0 | 定义为"企业级接口通用外挂"，对标 ESB | 用户明确产品是挂载在系统与ESB之间的适配层 |
| v3.0 | 明确双引擎定位：转换对标 Gateway，开发对标低代码 | 用户明确 PG 不是 ESB 替代品，而是互补关系 |
| v4.0 | **新增产品分级策略、部署形态、商业模式与市场分析** | 用户明确功能分级+多形态部署+商业化方向 |

## 七、产品分级与部署策略

### 7.1 产品分级：全量版 vs 简易版

PG 提供两个功能等级，客户根据实际环境选择：

| 维度 | 全量版 | 简易版 |
|------|--------|--------|
| 定位 | 正式生产环境，长期驻留服务 | 快速交付、现场实施、临时联调 |
| 配置库 | MySQL | 内嵌 H2 |
| 审计库 | MySQL（独立实例） | 内嵌 H2 |
| 缓存 | Caffeine + Redis 双层 | Caffeine 单层 |
| 业务数据库 | 支持多数据源动态切换 | 支持多数据源动态切换 |
| 报文转换引擎 | 完整 | 完整 |
| 可视化接口开发 | 完整 | 完整 |
| 监控审计 | 完整（独立审计库） | 基础（内嵌库记录） |
| 性能表现 | 高（双层缓存+分布式锁） | 中（单层缓存，适合中低频调用） |
| 适用场景 | 长期驻留银行服务器，作为正式接口服务 | 外包团队开发机调试、现场快速部署、同机部署 |

**核心原则：两个版本功能完全一致，差异仅在基础设施依赖。** 简易版不是"阉割版"，而是"零依赖版"。

### 7.2 部署形态：三种方式

| 形态 | 产物 | 场景 | 典型用户 |
|------|------|------|---------|
| Windows exe | 内嵌 JRE 的可执行文件 | 外包团队开发机、Windows 服务器 | 小型外包团队、现场实施 |
| Linux jar | 带 JRE 启动脚本的 fat jar | 部署到银行 Linux 服务器 | 所有外包/集成商 |
| 同机部署 | 部署在业务系统同一台机器上 | 绕过银行服务器审批流程 | 现场实施、维保场景 |

### 7.3 同机部署的价值

银行的服务器管理严格，开新端口、申请新机器的流程可能要走几周。同机部署直接绕过这些审批：

- 不需要向银行申请额外服务器资源
- 不需要走网络开通审批
- 不需要银行运维配合
- 半小时内部署完毕，当天交付

### 7.4 部署架构对比

```
简易模式（单机/同机部署）：
  PG (exe/jar)
    --内嵌 H2--> 配置库
    --内嵌 H2--> 审计库
    --Caffeine--> 单层缓存
    --直连--> 业务数据库(MySQL/Oracle/PG)
  零外部依赖，一个文件启动

标准模式（独立部署）：
  PG (jar)
    --MySQL--> 配置库
    --MySQL--> 审计库（独立实例）
    --Redis--> 双层缓存
    --直连--> 业务数据库(MySQL/Oracle/PG)
  需要外部 MySQL + Redis
```

### 7.5 技术改造要点

| 改造项 | 说明 | 优先级 |
|--------|------|--------|
| 内嵌数据库支持 | 引入 H2 作为配置库和审计库的替代，通过配置切换 | P0 |
| 缓存降级策略 | 无 Redis 时自动降级为 Caffeine-only 单层缓存 | P0 |
| 内嵌 JRE 打包 | Windows exe 内嵌 JRE（Launch4j/GraalVM），Linux 提供自带 JRE 的启动脚本 | P1 |
| 启动引导向导 | 首次启动时引导选择模式（简易/标准）并配置数据库连接 | P1 |
| 端口自动检测 | 启动时检测端口占用，自动选择可用端口 | P2 |

### 7.6 部署形态 vs 竞品对比

| 对比 | MuleSoft | Camel | YesApi | PG |
|------|----------|-------|--------|-----|
| 部署方式 | 需要集群+容器 | 需要嵌入应用 | 需要 Docker | **exe/jar 即用** |
| 能否同机部署 | 不能 | 不能 | 不能 | **能** |
| 能否零依赖启动 | 不能 | 不能 | 不能 | **能（简易版）** |
| 部署时间 | 天级 | 小时级 | 小时级 | **分钟级** |

| 医疗 | HIS/LIS/PACS 系统对接 | 报文转换 + 增量接口 |

### 8.3 市场规模估算

中国目前有约 **2400 家中小银行**（216 家城商行 + 1278 家农商行 + 564 家农信社 + 343 家村镇银行），每家背后至少 2-3 家 IT 外包/集成商，即 **5000+ 潜在客户**。金融外包市场年增速约 6%，低代码赛道预计到 2026 年超过 70% 的新应用将通过低代码平台构建。

### 8.4 商业模式：三阶段演进

**核心策略：用免费换用户、用买断验证付费、用年费换留存。**

#### 阶段一：免费期（0-12 个月）

- **目标**：获取种子用户，验证产品价值
- **策略**：功能免费使用，不收钱
- **限制**：接口数量上限 20 个（足够验证价值，但不够用）
- **支持**：不提供技术支持，仅提供社区文档
- **核心动作**：收集反馈、快速迭代、打磨体验
- **退出条件**：免费期最长 12 个月，到期后必须转买断或停止服务

#### 阶段二：一次性买断（12-24 个月）

- **目标**：建立收入，验证付费意愿
- **策略**：一次性买断制，永久使用当前版本
- **定价**：

| 版本 | 定价 | 包含内容 |
|------|------|---------|
| 简易版买断 | 2-3 万 | 内嵌数据库、单层缓存、基础功能 |
| 标准版买断 | 5-8 万 | MySQL+Redis、双层缓存、完整审计 |
| 企业版买断 | 10 万+ | 源码交付、定制开发、专属支持 |

- **核心动作**：功能稳定、文档完善、销售跟进
- **隐藏价值**：买断客户更容易转化为年费订阅对象

#### 阶段三：年费订阅（24 个月以后）

- **目标**：建立持续收入，支撑长期迭代
- **策略**：年费订阅，含版本更新 + 技术支持
- **定价**：2-3 万/年
- **核心动作**：重大功能升级、企业级支持、生态建设

#### 关键转折点

| 从免费到买断 | 从买断到年费 |
|-------------|-------------|
| 10+ 家活跃使用的外包公司 | 50+ 家付费客户 |
| 3+ 个成功案例（客户愿意背书） | 重大功能升级（如 AI 辅助、编排引擎） |
| 产品功能覆盖 80% 以上常见场景 | 企业级支持需求（SLA、专属客服） |
| 文档和教程完善，新人能独立上手 | 客户续费/升级意愿强 |

#### 年费转化话术

**对买断老客户：**
> "您买断的版本功能已经冻结了，最近我们上线了 AI 辅助接口生成和可视化编排引擎，这些功能只在年费版提供。年费 2 万/年，相当于您再雇一个初级开发半年的工资，但能持续获得所有新功能。"

**对新客户：**
> "买断 5 万用一辈子，但功能停在当前版本。年费 2 万/年，每年都能用最新功能，还有专属技术支持。建议先买断试用，满意后再转年费。"

#### 风险与应对

| 风险 | 说明 | 应对 |
|------|------|------|
| 免费期太长 | 用户习惯了免费，不愿付费 | 严格限制免费期时长和功能 |
| 买断客户不转年费 | 买断后"躺平"，不再付费 | 买断版功能冻结，重大功能只在年费版提供 |
| 年费定价太低 | 收入覆盖不了迭代成本 | 年费定价至少覆盖一个开发者的年薪 |
| 竞品免费 | 开源竞品（如 Juggle）免费 | PG 的差异化是"开箱即用+可视化"，不是功能多 |

### 8.5 "小而美"可行性

| 维度 | 判断 | 说明 |
|------|------|------|
| 市场规模 | 够养一个小团队 | 5000+ 潜在客户，渗透 2% = 100 家 |
| 竞争格局 | 有空白地带 | 巨头做重型 ESB，开源做编排，没人做"轻量可视化转换+CRUD"中间地带 |
| 技术壁垒 | 适中 | 报文转换+SQL Builder 不算高深，但组合在一起+可视化体验是壁垒 |
| 交付成本 | 极低 | exe/jar 单文件部署，远程半小时搞定 |
| 团队规模 | 2-3 人可维持 | 产品已基本成型，日常维护+小迭代足够 |

### 8.6 小而美的三条线

1. **不做大平台，做工具** — 类似 Postman，不改变客户工作流，只让某个环节更快。销售周期短、决策链路短、不需要招投标
2. **不做通用，做垂直** — 先吃透金融 IT 外包这一个场景，制造业/政务/医疗是延伸不是起点
3. **不做 SaaS，做私有部署** — 目标客群工作环境是银行内网，私有部署+离线可用，规避 SaaS 获客成本

### 8.7 风险与应对

| 风险 | 概率 | 应对 |
|------|------|------|
| 客户觉得"自己写代码也行" | 中 | 算账：一个接口 2 天 vs 10 分钟，10 个接口就是 20 人日 |
| 大厂做了类似功能 | 低 | 大厂做平台不做工具，PG 的轻量定位是护城河 |
| 外包公司自己造轮子 | 中 | 开源核心模块，让造轮子变得不划算 |
| 银行信创要求 | 高 | 支持 JDK 17 + 国产数据库（达梦/人大金仓/TiDB/OceanBase）+ 国产中间件（Tendis/Lindorm）+ 国产操作系统（麒麟/统信）+ 国密算法（SM2/SM3/SM4），信创适配作为第一次优化重点 |

## 十、核心结论

### 10.1 产品定位总结

PowerGateway 是一个**双引擎合一的企业级接口通用外挂**：

- **引擎一（报文转换）**：替代 Gateway 适配层，帮 ESB 干脏活，让业务系统零代码接入
- **引擎二（接口开发）**：替代低代码接口开发，10 分钟配好一个增量接口

PG 与 ESB 是**互补关系**，不是替代关系。

### 10.2 竞争策略

| 对比对象 | PG 的策略 | 话术 |
|----------|----------|------|
| vs MuleSoft/Camel | 轻量替代 | "不需要 ESB 团队，新手也能搞定接口转换" |
| vs YesApi/Juggle | 双引擎合一 | "别人只做接口开发或只做编排，我们转换+开发都能干" |
| vs 国产 ESB | 互补定位 | "不是替代你们的 ESB，是帮你们省 50% 的适配工作量" |
| vs 直接改代码 | 零侵入 | "不改业务系统一行代码，外挂式接入" |
| vs 外包/驻场开发 | 降本增效 | "不用找外包团队，自己人 10 分钟配好一个接口" |

### 10.3 最紧迫的短板

**服务注册发现（Nacos + Eureka）** — 企业内网已有微服务生态，PG 发布的接口必须能被注册中心发现，否则就是一个孤岛系统。金融行业老系统大量使用 Eureka，不能只支持 Nacos。这是接入企业技术栈的入场券。
