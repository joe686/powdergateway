# PowerGateway 测试工具计划

> 文档日期：2026-06-20  
> 目标：构建一套自包含的自动化测试工具，解决 PG 无上下游系统无法端到端测试的问题

---

## 零、测试工作流（AI 分工）

### 核心原则

> **我承担全部测试任务** — 管理台 UI 浏览器操作、后端 API 调用验证、数据库状态校验、日志检查，全部由我完成。  
> **开发者 AI 专注修复** — 只根据我的测试报告定位和修改代码，不参与测试执行。  
> **回归闭环** — 修复后必须重新执行全部相关测试用例，确认无回归问题才算闭环。

| 原则 | 说明 |
|------|------|
| **我全测** | 管理台 UI 操作（browser 工具）、后端 API（HTTP 调用）、数据库校验（SQL 查询）、日志检查（Grep/Read），全部由我执行 |
| **只记不改** | 测试过程中发现的问题，记录到问题清单，不修改任何代码 |
| **开发者 AI 只改不测** | 根据我的测试报告定位并修复 BUG，不负责执行测试 |
| **回归闭环** | 修复后必须全部重新验证，确认 BUG 已修复且无副作用才算闭环 |

### 工作流

```
┌─────────────────────────────────────────────────────────────────┐

                            │

│  - 根据测试报告中的失败详情定位 BUG                                │

│  └─ 输出：修复后的代码 + 修复说明                                 │

                            ▼ 修复完成
┌─────────────────────────────────────────────────────────────────┐
│  步 骤 3：我（TRAE AI）回归验证                                    │
│  - 重新执行步骤 1 中全部失败用例                                   │
│  - 同时执行关联用例，确认修复无副作用                               │
│  └─ 输出：回归验证报告（全部通过 / 仍有失败）                      │
└─────────────────────────────────────────────────────────────────┘
```

## 一（补充）、测试环境要求与前置检查

### 1.1 环境依赖清单

| 组件 | 版本要求 | 用途 | 检查命令 |
|------|---------|------|---------|

| Node.js | 16+ | 前端构建（如需重新构建） | `node -v` |

### 1.2 数据库初始化检查

启动应用前必须确认以下数据库和表已就绪：

**配置库** `powergateway_config`：
- `sys_user` — 用户表（需有默认管理员账号）
- `sys_config` — 系统配置表（需有 8 条默认配置）
- `convert_template` — 转换模板表
- `channel_config` — 渠道配置表
- `port_route` — 端口路由表
- `db_connection` — 数据库连接表
- `interface_config` — 接口配置表
- `shard_config` — 分库分表配置表
- `field_formula` — 字段公式表
- `sys_log` — 操作日志表
- `sys_log_history` — 操作日志归档表
- `perf_stat` — 性能统计表
- `perf_alert` — 告警记录表

**审计库** `powergateway_audit`：
- `sql_audit_log` — SQL 审计日志表

> **注意**：`init.sql` 中审计表 DDL 写在配置库脚本内，实际运行时会根据 `application.yml` 中的 `audit` 数据源配置写入审计库。若审计库表缺失，应用启动会报错。

### 1.3 启动前自检清单

```
□ MySQL 服务已启动（服务名：MySQL80）
□ Redis 服务已启动（服务名：Redis）
□ powergateway_config 数据库存在且表结构完整
□ powergateway_audit 数据库存在且 sql_audit_log 表已创建
□ application.yml 中数据库连接信息正确（URL/用户名/密码）
□ 8080 端口未被占用
□ 前端 dist 目录已构建（npm run build 已执行）
```

### 1.4 应用启动方式

```bash
# 方式一：Maven 启动（开发调试）
cd backend
mvn spring-boot:run -DskipTests

# 方式二：运行打包后的 jar（生产/测试环境）
cd backend
mvn clean package -DskipTests
java -jar target/backend-1.0.0-SNAPSHOT.jar

# 方式三：IDE 中直接运行 PowergatewayApplication 主类
```

### 1.5 前端访问方式

前端为静态资源，已构建到 `frontend/dist/` 目录。有以下访问方式：

| 方式 | 说明 |
|------|------|
| 独立部署 | 将 `frontend/dist/` 内容部署到 Nginx/Apache，配置反向代理到后端 8080 端口 |
| 开发模式 | `cd frontend && npm run dev`，访问 `http://localhost:5173` |
| 直接打开 | 用浏览器打开 `frontend/dist/index.html`（部分 API 调用可能因跨域失败，仅适合查看静态页面） |

> **测试建议**：使用开发模式 `npm run dev` 启动前端，可以实时看到修改效果，且已配置好代理到后端 8080 端口。

### 2.3 数据库校验操作规范

#### 校验时机

| 测试步骤 | 校验内容 |
|---------|---------|
| 保存操作后 | 查询对应表，验证数据是否写入 |
| 删除操作后 | 验证逻辑删除（`deleted=1`）或物理删除 |
| 更新操作后 | 验证字段值是否正确更新 |
| 发布操作后 | 验证 `status` 字段状态流转 |

#### 常用校验 SQL

```sql
-- 验证模板保存
SELECT * FROM convert_template WHERE name = '测试模板' ORDER BY id DESC LIMIT 1;

-- 验证操作日志记录
SELECT * FROM sys_log WHERE module = '模板管理' ORDER BY id DESC LIMIT 5;

-- 验证 SQL 审计记录
SELECT * FROM sql_audit_log WHERE interface_id = 123 ORDER BY id DESC LIMIT 1;

-- 验证性能统计记录
SELECT * FROM perf_stat WHERE interface_id = 123 ORDER BY id DESC LIMIT 1;

-- 验证用户密码加密
SELECT password FROM sys_user WHERE username = 'testuser';
-- 预期：以 $2a$ 开头的 BCrypt 哈希值

-- 验证数据库连接密码加密
SELECT password FROM db_connection WHERE name = '测试MySQL';
-- 预期：AES 加密后的密文（非明文）
```

### 2.4 日志检查操作规范

#### 日志文件位置

| 环境 | 日志位置 |
|------|---------|
| Maven 启动 | 控制台输出 |
| Jar 运行 | `backend/logs/` 目录（若配置了 logback） |
| IDE 运行 | 控制台输出 |

#### 检查要点

```
□ 无 ERROR 级别日志
□ 无异常堆栈（Exception/StackTrace）
□ 审计日志异步写入无阻塞主线程
□ 性能统计记录生成正常
□ SQL 执行无慢查询警告
```

#### 关键日志关键词

| 关键词 | 含义 | 预期 |
|--------|------|------|
| `SqlAuditAspect` | SQL 审计切面 | 每次接口调用都应出现 |
| `PerfStatAspect` | 性能统计切面 | 每次接口调用都应出现 |
| `SysLogAspect` | 操作日志切面 | 每次管理台操作都应出现 |
| `ERROR` | 错误日志 | 不应出现 |
| `Exception` | 异常 | 不应出现 |
| `Slow query` | 慢查询 | 不应出现 |

## 一、问题分析

### 1.1 当前测试能力

PG 已有 **280+ 个单元/集成测试**，全部基于 H2 内存库 + MockMvc，覆盖了全部核心功能模块。

| 模块 | 测试类数 | 测试方法数 | 测试方式 |
|------|---------|-----------|---------|
| P0 基础平台 | 3 | ~20 | H2 内存库 + MockMvc |
| M1 报文转换 | 11 | ~90 | 纯单元测试 + MockMvc |
| M2 数据接口 | 17 | ~100 | H2 内存库 + MockMvc |
| SYS 系统管理 | 11 | ~40 | H2 内存库 + MockMvc |
| AUX 辅助功能 | 1 | ~12 | MockMvc |

### 1.2 当前测试的局限

| 局限 | 说明 |
|------|------|
| **无真实数据库测试** | M2 系列使用 H2 内存库，无法验证真实 MySQL/Oracle/PG 的 SQL 兼容性 |
| **无上下游系统** | 报文转换（M1）的端口分发功能使用 MockRestServiceServer 模拟 B 系统，无法验证真实网络调用 |
| **无端到端链路** | 每个单元独立测试，无法验证"配置→发布→调用→审计→监控"的完整链路 |
| **AI 无法参与测试** | 测试代码是静态的，AI 无法动态添加测试用例或调用工具验证功能 |

### 1.3 现有测试工具清单

| 工具/类 | 位置 | 用途 |
|---------|------|------|
| H2 内存库 | `application-test.yml` | 替代 MySQL，测试环境零依赖 |

## 二、测试工具架构设计

### 2.1 三层测试体系

```
第一层：单元测试（已有，280+ 个）
  H2 内存库 + MockMvc，验证单个功能点的正确性
  适合：CI/CD 每次提交自动运行

第二层：集成测试工具（新增）
  内嵌真实数据库 + Mock 上下游，验证完整链路
  适合：发布前验证、AI 自动补充测试

第三层：端到端测试工具（新增）
  内置 HTTP 服务器模拟上下游，验证真实网络调用
  适合：试点前验证、客户现场演示
```

### 2.2 集成测试工具：PG-TestKit

PG-TestKit 是一个**独立的测试工具项目**，与 PG 项目本体（`backend/`、`frontend/`）平级，位于仓库根目录的 `pg-testkit/`，不依赖 backend 源码，通过 HTTP 调用 PG 后端、通过 JDBC 直连配置库/审计库完成校验。提供以下能力：

> **独立性原则**（CHG-008 调整）：
> - 测试工具源码、依赖、构建产物均与项目本体物理分离，不放入 `backend/src/test/`
> - 测试工具作为独立 Spring Boot 应用启动（默认端口 8081），不与后端 8080 端口冲突
> - 测试工具不引用 backend 内部类，仅通过 REST API + JDBC 与 PG 交互，避免耦合
> - backend 的 `pom.xml` 不引入 pg-testkit 依赖，反之 pg-testkit 也不依赖 backend jar

#### 能力一：内嵌数据库环境

| 功能 | 说明 |
|------|------|
| 自动启动内嵌数据库 | 使用 Testcontainers 自动拉起 MySQL/PostgreSQL Docker 容器（有 Docker 时），或使用 H2 MySQL 模式（无 Docker 时） |
| 自动建表和灌数据 | 基于 init.sql 自动建表，灌入测试数据（用户、模板、字典等） |
| 多数据库类型切换 | 支持在 MySQL/PostgreSQL/H2 之间切换测试，验证 SQL 兼容性 |
| 测试数据工厂 | 提供 `TestDataFactory` 工具类，快速生成接口配置、转换模板、字段映射等测试数据 |

#### 能力二：Mock 上下游系统

| 功能 | 说明 |
|------|------|
| 内置 HTTP Mock 服务器 | 基于 Spring Boot 内嵌 Tomcat 或 WireMock，模拟 B 系统接收请求并返回预设响应 |
| 响应模板 | 支持配置多种响应模板（成功/失败/超时/格式错误），模拟各种异常场景 |
| 请求验证 | 自动记录 PG 发出的请求（URL、Header、Body），验证报文转换的正确性 |
| 延迟模拟 | 支持配置响应延迟（0ms/100ms/1000ms/5000ms），验证超时处理 |

#### 能力三：完整链路测试

| 功能 | 说明 |
|------|------|
| 全链路测试框架 | 提供 `IntegrationTestTemplate` 基类，封装"登录→配置→发布→调用→验证审计→验证监控"完整流程 |
| 断言工具集 | 提供 `PgAssertions` 工具类，封装常用断言（响应格式、审计记录、性能统计等） |

### 2.3 端到端测试工具：PG-MockServer

PG-MockServer 是一个独立的轻量 HTTP 服务器，模拟 PG 的上下游系统。

#### 架构

```
测试流程：
  PG-MockServer(模拟B系统) <--HTTP--> PowerGateway <--HTTP--> 测试客户端(模拟A系统)

或者：
  测试客户端 --> PowerGateway --动态SQL--> 内嵌数据库(模拟业务库)
```

#### 功能清单

| 功能 | 说明 |
|------|------|
| 接收任意 HTTP 请求 | 记录请求的 URL、Method、Header、Body |
| 返回预设响应 | 按配置返回 JSON/XML/纯文本响应 |
| 响应延迟控制 | 可配置固定延迟或随机延迟 |
| 请求匹配规则 | 按 URL 路径、Header、Body 内容匹配不同的响应 |
| 请求历史查询 | 提供查询 API，查看所有收到的请求记录 |
| 报文格式验证 | 自动验证收到的报文是否符合预期格式（JSON Schema / XPath） |

### 6.1 与项目本体的交互方式

pg-testkit 作为独立项目，通过以下方式与 PG 项目本体交互，**不引用 backend 任何内部类**：

| 交互方向 | 方式 | 说明 |
|---------|------|------|
| 测试工具 → PG 后端 | HTTP REST | 调用 `/api/login`、`/api/convert`、`/api/exec/{id}` 等接口，使用 RestTemplate |
| 测试工具 → 配置库 | JDBC | 直连 `powergateway_config` 库，校验 `convert_template`、`interface_config` 等表数据 |
| 测试工具 → 审计库 | JDBC | 直连 `powergateway_audit` 库，校验 `sql_audit_log` 表数据 |
| PG 后端 → 测试工具 Mock | HTTP | PG 端口分发（`/api/dispatch`）转发请求到 Mock 服务器（默认 9999 端口） |

### 6.2 启动顺序

```
1. 启动 MySQL（powergateway_config + powergateway_audit 库已初始化）
2. 启动 Redis
3. 启动 PG 后端：cd backend && mvn spring-boot:run          # 端口 8080
4. 启动测试工具：cd pg-testkit && mvn spring-boot:run        # 端口 8081
5. AI 通过 /test/* API 控制测试工具，或直接运行 E2E 测试
```
