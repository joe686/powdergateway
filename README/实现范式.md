# 核心实现范式

## 后端模式

```java
// 0. 统一响应/异常：所有接口必须返回 Result<T>（common/Result.java），
//    业务校验失败抛 BusinessException(code, message)，由 GlobalExceptionHandler 统一处理
// 1. 格式转换：各格式先统一解析为 Map<String, Object>，再序列化
FormatConverter.convert(src, FormatType.JSON, FormatType.XML)

// 2. 字段加工：策略模式，规则列表顺序执行
interface FieldProcessStrategy { String process(String value, Map<String, String> params); }
// 实现：TrimProcessor / PadProcessor / SubstringProcessor / CaseProcessor / TypeCastProcessor

// 3. SQL生成：Builder模式，config_json → SQL（不存硬编码SQL）
QueryBuilder / InsertBuilder / UpdateBuilder / DeleteBuilder

// 4. 动态数据源：dynamic-datasource-spring-boot-starter，按 dbId 切换
// 5. 审计日志：AOP @Around 拦截 Executor，LinkedBlockingQueue + @Async 异步写审计库
// 6. 双层缓存：Caffeine → Redis → DB，分布式锁防击穿（SET NX PX 3000）
// 7. 接口发布后统一执行入口：POST /api/exec/{interfaceId}
```

## 前端模式

```js
// 1. 配置步骤用 el-steps，接口向导统一用 SYS-5 的 InterfaceWizard.vue
// 2. 字段映射用 vue-draggable-next 拖拽
// 3. 所有列表页：搜索栏 + el-table + el-pagination + 导出按钮（统一样式）
// 4. 删除操作必须用 el-popconfirm 二次确认
// 5. 步骤向导中间数据存 Pinia wizardStore，同步 localStorage 防丢失
```

## 每个单元标准结构

编写代码时，每个交付单元须包含：范围（含）、不含（排除项）、前置依赖、验收标准，与 `README/需求拆分与最小实现方案.md` 对应。
