## 根因分析与预防措施

### 根因：前后端开发缺少交叉验收机制

#### 直接原因
开发 M1-7 时，专注于后端逻辑实现（Controller/Service/Mapper），在后端通过测试后认为单元完成，未同步检查前端页面是否配套完成。前端的菜单、路由、页面、API 封装四个文件均依赖手动新增，属于容易遗漏的"无代码提示"工作。

#### 深层原因
1. **单元验收标准执行不彻底**：`需求拆分与最小实现方案.md` 中 M1-7 的验收标准明确要求前端配置页面，但验收时只运行了后端接口测试，未通过浏览器实际访问验证
2. **前后端是独立工程，没有强依赖关系**：后端测试通过不代表前端已完成，两者需独立核查
3. **没有"交付检查清单"**：每个单元应有标准化的交付清单（后端接口 + 前端页面 + 菜单入口 + 文档更新），现在缺少这个约束

### 预防措施

#### 措施一：每个单元完成时执行标准交付检查清单
每个含前端功能的单元，提交前必须逐项确认：

| 检查项 | 说明 |
|--------|------|
| ☐ 后端接口 | Controller 接口通过 Swagger 可访问 |
| ☐ 前端 API 封装 | `src/api/xxx.js` 已创建，方法名与后端接口一一对应 |
| ☐ 前端页面 | `src/views/xxx/Xxx.vue` 已创建，含列表/表单/操作 |
| ☐ 路由注册 | `src/router/index.js` 已添加对应路由 |
| ☐ 菜单入口 | `SideMenu.vue` 已添加菜单项，点击可跳转页面 |
| ☐ 浏览器验证 | 在 `localhost:5173` 实际操作一遍：打开菜单→新增→保存→列表刷新→删除 |
| ☐ 需求文档更新 | `需求拆分与最小实现方案.md` 和 `开发计划.md` 标记完成状态 |

#### 措施二：在 CLAUDE.md 中明确前端单元完成定义
约定凡是有前端交互的单元，"完成"的定义必须包括"浏览器可操作"，而不仅仅是"后端测试通过"。

#### 措施三：单元开发前先在前端创建占位页面
开发后端之前，先在前端把路由和菜单占位建好（哪怕用 `PlaceholderView`），这样在菜单上就能看到这个功能的入口，后续替换为真实页面时不会遗漏。

#### 措施四（问题3/4/5 补充）：用户交互必须经过 UX 验收，不能只看代码存在

| 易犯错误 | 正确做法 |
|---------|---------|
| 表单输入用自动生成默认值代替用户输入 | 凡用户需要命名的字段，必须提供对话框/输入框让用户填写 |
| 自动触发 API 的操作未检查前置条件 | 触发 API 前验证必要输入是否就绪（如预览接口需要 inputValue 非空） |
| 内部计数器 id 当作业务标识 | 需要被引用的实体必须设计用户可见、可自定义的 `code`/`name` 字段 |

### BUG-009：PerfStatAspect 主线程同步查询接口配置增加 DB 开销（已解决）
- **发现时间**：2026-06-20（代码审查发现，与开发计划 OPT-15 关联）
- **解决时间**：2026-06-20
- **发现人**：TRAE AI（测试）
- **测试用例**：代码审查 / SYS2PerfStatAspectTest
- **问题描述**：`PerfStatAspect.java` 第 34 行在切面主线程中执行 `interfaceConfigMapper.selectById(interfaceId)` 查询数据库，用于获取 `opType`。这违反了审计架构"旁观者"原则，增加了主线程 DB 开销，在高并发场景下可能成为瓶颈。
- **根因分析**：性能统计切面需要在统计记录中写入 `opType`（SELECT/INSERT/UPDATE/DELETE），但当前实现是在主线程同步查询数据库获取，而非从事先缓存的配置中读取。
- **修复内容**（OPT-15 配置热缓存）：
  - 新增 `InterfaceOpTypeCache` 组件：基于 Caffeine 的本地缓存，TTL 1 小时，最大容量 5000
  - `PerfStatAspect` 改为从 `InterfaceOpTypeCache.getOpType()` 读取，缓存未命中时回源查询并回填
  - `InterfaceConfigService.publish()` 发布接口时调用 `preload()` 预加载 opType 到缓存
  - `InterfaceConfigService.disable()` / `delete()` 禁用/删除接口时调用 `evict()` 清除缓存
  - 缓存未命中时同步回源查询并回填（首次调用场景），高频接口缓存命中率接近 100%

### BUG-012：用户表 `create_time` 字段为 null（已解决）

- **发现人**：TRAE AI（测试）
- **测试用例**：SYS3UserTest / API 测试
- **问题描述**：`GET /api/user/list` 返回的用户数据中，`createTime` 字段为 null
- **根因分析**：`DataInitializer.java` 创建 admin 用户时未设置 `createTime` 字段，虽然数据库表定义了 `DEFAULT CURRENT_TIMESTAMP`，但 MyBatis-Plus 插入时传入了 null 值覆盖了默认值。实体类虽有 `@TableField(fill = FieldFill.INSERT)` 注解，但缺少 `MetaObjectHandler` 实现导致注解不生效。

  - 新增 `MyMetaObjectHandler` 组件，实现 MyBatis-Plus `MetaObjectHandler` 接口，配合 `@TableField(fill = FieldFill.INSERT)` 注解自动填充 `createTime`/`updateTime`
- **影响文件**：
  - `backend/src/main/java/com/powergateway/config/DataInitializer.java`（显式设置时间）
  - `backend/src/main/java/com/powergateway/config/MyMetaObjectHandler.java`（新增，自动填充处理器）
