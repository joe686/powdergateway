## 第一次优化（功能迭代，6-12 个月）

基于竞品分析报告 v4.0 确定的方向，进行第一次功能迭代优化。

**最小可用边界：当前 PG + OPT-1（Nacos+Eureka 注册发现）。** 做完 OPT-1 即可拿出去试点验证，其余优化根据试点反馈排优先级。

试点部署方式：在客户现有 MySQL 实例中建一个 schema 给 PG 用，复用客户已有的 Redis，拷贝 jar + 改配置 + 启动，四步完成部署。

### 试点后根据反馈迭代

| 单元 | 优先级 | 内容 | 说明 |
|------|--------|------|------|
| **OPT-3** | P1 | Mock 服务 | 基于接口配置自动生成 Mock 数据，支持前后端并行开发 |
| **OPT-14** | P1 | 数据字典管理 + 字典转换策略 | 新增全局字典管理模块，维护字典类型（性别/状态/币种/机构类型等）及源值→目标值映射关系。字段加工引擎（M1-3）新增"字典转换"策略类型，配置接口时选择字典即可自动完成值映射。字典数据加载到 Caffeine 本地缓存（字典变更时主动刷新），运行时直接查缓存转换，零数据库开销。字典变更时所有引用接口自动生效，无需逐个修改 |
| **OPT-15** | P1 | 配置热缓存（接口配置 + 转换模板） | 接口配置（interface_config）和转换模板（convert_template）发布后，将解析后的配置对象（SQL 模板、字段映射、加工规则、渠道路由等）预加载到 Caffeine 本地缓存。执行时直接使用缓存对象，不查库不解析 JSON。配置变更时主动刷新缓存（基于 ApplicationEvent 通知），保证一致性。压测场景下消除数据库查询和 JSON 解析开销，大幅降低 P99 延迟 |
| **OPT-16** | P1 | 报文格式增强 — JSON/XML 复杂结构支持 | 增强转换模板对复杂报文格式的支持能力，JSON 和 XML 双向覆盖。JSON 增强：(1) 数组根节点处理（`[{...}]` 格式）；(2) 多层嵌套结构映射（任意深度嵌套路径）；(3) 循环/重复结构（数组内对象批量映射）；(4) 特殊字符转义（引号、换行等）。XML 增强：(1) 多层嵌套标签映射（任意深度 XPath 路径）；(2) 循环/重复节点（同名子节点批量映射）；(3) 属性与子元素区分处理（`<item id="1">` 中 id 为属性、内容为子元素）；(4) 命名空间支持（xmlns 前缀处理）；(5) CDATA 段处理（含特殊字符的文本节点）；(6) XML 声明头配置（`<?xml version="1.0" encoding="GBK"?>` 编码和版本可配）。通用增强：(1) 长度标识字段（报文体前自动计算填充长度，如 `LEN=0123`）；(2) 标准报文头模板（交易码、版本号、时间戳、签名等固定格式报文头段）。前端映射配置界面同步优化，支持 JSON/XML 嵌套结构的可视化预览和路径选择 |
| **OPT-17** | P1 | 响应格式模板 — JSON/XML 扁平/嵌套/自定义 | 接口配置（M2 系列）新增响应格式模板功能，JSON 和 XML 双向覆盖。提供三种基础响应模型：(1) 扁平化响应 — 查询结果直接展平为单层结构，JSON 如 `{"name":"张三","age":30}`，XML 如 `<name>张三</name><age>30</age>`；(2) 嵌套响应 — 按配置的层级结构组织，JSON 如 `{"data":{"user":{"name":"张三"}},"code":"0000"}`，XML 如 `<response><code>0000</code><data><user><name>张三</name></user></data></response>`；(3) 自定义模板 — 可视化配置响应结构，支持固定字段（code/msg/total）+ 动态数据字段组合，JSON 和 XML 结构各自独立配置。提供常用响应模板库（银行标准响应、ESB 标准响应等），可保存为自定义模板复用。同一接口可配置多个响应格式，运行时通过参数或渠道码选择 JSON 或 XML 不同格式输出 |

| **OPT-5** | P1 | OpenAPI 规范导出 | 支持导出标准 OpenAPI 3.0 规范，方便接入 APISIX/Spring Cloud Gateway |

## 缓存策略

Caffeine（JVM 本地缓存，纳秒级）和 Redis（分布式缓存，毫秒级）互补使用，按数据类型选择合适的缓存层级。

| 数据类型 | 缓存策略 | 原因 |
|----------|---------|------|
| 接口配置（SQL模板、字段映射） | Caffeine 单层 | 体积小、访问极频繁、单实例用不需要共享 |
| 转换模板（字段映射、加工策略） | Caffeine 单层 | 同上 |
| 数据字典（源值→目标值映射） | Caffeine 单层 | 几乎不变、几百条数据、纳秒级响应 |
| 渠道路由规则 | Caffeine 单层 | 变化少、访问频繁 |
| 表结构元数据（M2-2） | Redis 单层 | 体积可能较大、多实例可共享 |
| 接口查询结果（M2-10） | Caffeine + Redis 双层 | 热点数据本地加速 + 多实例共享防重复查库 |

**选择原则：**
- **配置类数据**（字典、模板、路由）→ Caffeine 就够，不需要 Redis
- **业务查询结果**（接口返回数据）→ Caffeine + Redis 双层，本地加速 + 多实例共享
- **元数据**（表结构）→ Redis，体积大且可共享
- **缓存刷新**：配置变更时通过 ApplicationEvent 主动刷新 Caffeine 缓存，保证一致性

## 审计架构原则

**审计是旁路观察者，不是交易参与者。** 审计功能（SQL 审计、操作日志、性能统计）必须满足以下原则：

1. **不阻塞主链路** — 所有审计写入必须异步（LinkedBlockingQueue + 守护线程），队列满时静默丢弃
2. **不增加主链路开销** — 审计切面中不得有任何同步数据库查询/网络调用
3. **不影响业务结果** — 审计失败（写入异常、队列溢出）不得导致业务接口报错或回滚
4. **仅做事后记录** — 审计是事后监督和审查工具，不参与交易发起、权限校验、结果判定等业务逻辑

### 架构变更：三库合并为单库

**当前架构**：配置库（master）+ 审计库（audit）两个 MySQL 数据源，通过 `@DS("audit")` 路由。

**目标架构**：合并为单个数据库，通过表名前缀区分职责：
- `sys_*` — 系统配置表（用户、权限、系统配置等）
- `audit_*` — 审计日志表（SQL 审计、操作日志、性能统计等）
- 业务表（接口配置、转换模板、渠道配置等）无前缀

**变更范围**：
- `application.yml`：移除 `audit` 数据源配置，只保留 `master`
- `SqlAuditLogMapper.java`：移除 `@DS("audit")` 注解
- `AuditLogService.java`：确认无数据源路由依赖
- `init.sql`：将审计表和配置表合并到同一个建库脚本中
- 业务库（动态连接的目标数据库）保持不变，仍然是运行时动态切换

**收益**：客户部署时只需建一个数据库，降低部署复杂度，符合"单库即可运行"的产品定位。

### 审计数据归档规约

单库架构下，审计表数据会持续增长，必须通过定期归档/清理防止单表膨胀影响整体性能。

| 审计表 | 保留策略 | 归档方式 | 说明 |
|--------|---------|---------|------|
| `audit_sql_log` | 保留 90 天 | 超期归档到 `audit_sql_log_history`，归档后删除原表数据 | SQL 审计日志量大，需定期归档 |
| `audit_sys_log` | 保留 90 天 | 已有 `SysLogArchiveJob` 归档到 `audit_sys_log_history`，保持不变 | 操作日志已有归档机制 |
| `audit_perf_stat` | 保留 30 天 | 超期直接清理（DELETE），不归档 | 性能统计数据时效性短，无需长期保留 |

**实现方式**：定时任务（Spring `@Scheduled`），每天凌晨执行，归档/清理天数通过 `sys_config` KV 配置，支持在线调整。
