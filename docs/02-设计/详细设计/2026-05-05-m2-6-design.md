# M2-6 删除接口配置 设计规格

**日期**：2026-05-05  
**单元**：M2-6  
**前置依赖**：M2-1（数据库连接管理）、M2-2（表结构查询）

---

## 一、范围

可视化配置 DELETE 接口，包含：
- 多表条件配置（最多 3 张表，每张表独立配置 WHERE 条件）
- 删除预览（SELECT 前 10 条待删数据）
- 批量删除保护开关（`allow_batch_delete`）
- 删除前记录各表 COUNT，写入审计日志
- 多表事务执行（任意表失败全部回滚）

**不含**：接口发布（M2-7）、删除审批流程（保证外部调用及时性）

---

## 二、数据结构

### `DeleteConfigJson`（config_json 存储格式）

```json
{
  "tables": [
    {
      "tableName": "orders",
      "conditions": [
        {"field": "status", "op": "EQ", "paramKey": "status"}
      ]
    },
    {
      "tableName": "order_items",
      "conditions": [
        {"field": "order_id", "op": "EQ", "paramKey": "orderId"}
      ]
    }
  ]
}
```

每张表内嵌自己的 `conditions`，不依赖 tableName 字段区分。条件操作符：`EQ / NE / GT / LT / LIKE`。

### `InterfaceSaveRequest` 扩展

新增字段 `allowBatchDelete`（Integer，0=否，1=是，默认 0），保存时写入 `interface_config.allow_batch_delete`。

## 三、后端设计

### 3.1 新增文件

#### `DeleteConfigJson.java`
```java
@Data
public class DeleteConfigJson {
    private List<TableDeleteConfig> tables;

    @Data
    public static class TableDeleteConfig {
        private String tableName;
        private List<ConditionItem> conditions;
    }

    @Data
    public static class ConditionItem {
        private String field;
        private String op;      // EQ / NE / GT / LT / LIKE
        private String paramKey;
    }
}
```

#### `DeleteBuilder.java`
```java
public class DeleteBuilder {
    public static SqlResult build(String tableName,
                                   List<ConditionItem> conditions,
                                   Map<String, Object> params) {
        // DELETE FROM tableName WHERE field op ?
        // 条件为空时抛 IllegalArgumentException
    }
}
```

### 3.2 扩展 `InterfaceConfigService`

#### `deletePreview(Long id, Map<String, Object> params)`
- 解析 `config_json` → `DeleteConfigJson`
- 对每张表执行 `SELECT * FROM tableName WHERE {conditions} LIMIT 10`
- 返回 `Map<String, List<Map<String, Object>>>`，key 为 tableName

#### `executeDelete(Long id, Map<String, Object> params)`
1. 解析配置，获取各表条件
2. 对每张表执行 `SELECT COUNT(*) WHERE {conditions}`，汇总 `totalCount`
3. 若 `totalCount > 1` 且 `allowBatchDelete = 0` → 抛 `BusinessException(400, "批量删除保护：待删记录共 N 条，请先开启允许批量删除")`（错误信息含各表条数）
4. 多表 JDBC 手动事务：依次执行各表 DELETE，任意失败全部回滚
5. 审计日志：`sqlText` 记录实际 DELETE SQL，`beforeSnapshot` 记录各表 COUNT 信息（`{"orders": 3, "order_items": 5}`）

标注 `@AuditLog`，复用 M2-9 审计链路。

### 3.3 扩展 `InterfaceConfigController`

新增接口：
```
POST /api/interface/{id}/delete-preview
Body: { "params": {...} }
Response: Result<Map<String, List<Map<String, Object>>>>
```

`execute` 方法增加 `DELETE` 分支，调用 `executeDelete`，设置 `AuditContext`。

### 3.4 扩展 `InterfaceSaveRequest`

```java
/** 是否允许批量删除：0=否，1=是，null=不修改（默认0） */
private Integer allowBatchDelete;
```

`save()` 方法中：若 `type=DELETE` 且 `allowBatchDelete != null` 则写入 entity。

## 四、前端设计

### 4.1 `DeleteConfig.vue`（3步向导）

**Step 1 — 基本信息**
- 接口名称（el-input，必填）
- 数据库连接（el-select，必填，变更时重新加载表结构）
- 允许批量删除（el-switch，关闭时显示橙色警告"关闭后单次删除超过1条将被拒绝"）

**Step 2 — 多表条件配置**
- 「+ 添加表」按钮，最多 3 张表
- 每张表：下拉选表名 + 「删除」按钮
- 每张表下方用 `<ConditionBuilder>` 配置 WHERE 条件（`showTableColumn=false`，因为每表独立）
- 需至少一张表且至少一个条件才能进入下一步

**Step 3 — 预览 + 保存**
- 配置摘要（el-descriptions：接口名、表数量、各表条件数）
- 「预览待删数据」按钮 → el-dialog 弹窗，调用 `deletePreview` API，分表展示前 10 条（el-tabs 按表名切换）
- 「保存」按钮 → 调用 `saveInterface`，成功后跳转 `/interface/dev`

### 4.2 `api/interface.js` 新增

```js
export const deletePreview = (id, params) =>
  request.post(`/interface/${id}/delete-preview`, { params })
```

### 4.3 路由注册

在 `src/router/index.js` 将 `/interface/delete-config` 对应组件从 `PlaceholderView` 替换为 `DeleteConfig.vue`。

## 五、测试计划

### `M26DeleteBuilderTest`（纯单元测试）

- 单表单条件 EQ → SQL 结构正确
- 多条件 AND 连接 → 正确
- LIKE 操作符 → 正确
- conditions 为空 → 抛 `IllegalArgumentException`

### `M26DeleteConfigServiceTest`（@SpringBootTest + H2）

- `deletePreview`：返回前10条，结果 Map 按 tableName 分组
- `executeDelete`：批量保护拦截（count=2，allowBatchDelete=0 → 抛异常）
- `executeDelete`：开启批量删除后正常执行，返回影响行数
- `executeDelete`：审计日志记录 COUNT 信息

## 六、验收标准

1. 配置2张表的 DELETE 接口，预览功能分表显示前10条待删数据
2. `allowBatchDelete=0` 时，待删记录 > 1 条返回错误且包含各表条数
3. `allowBatchDelete=1` 时，正常执行多表 DELETE，事务正确（其中一张表失败时均回滚）
4. 审计日志中可查到删除前各表 COUNT 数据
5. 前端 Step 2 至少需要一张表 + 一个条件才能进入 Step 3
