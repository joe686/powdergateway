## 一、目标

把 `配置导入/导出` 和 `接口文档` 两个功能打通为**一体化的文档 ↔ 配置双向管道**：

```
   配置                      文档                         配置
  （DB）    ── 导出 ──>   （md/excel）  ── 编辑 ──>   （DB）
              ↑                                        ↓
             生成                                    导入
```

**用户价值**：
1. 一份 md/excel 既是可读文档，也是可编辑的配置源
2. 客户/业务方可以在 excel 里改配置，回传后一键导入生效
3. 版本控制友好（md 可以 git diff）

---

## 二、导出格式对比

| 格式 | 用途 | 优点 | 缺点 | 是否可反导入 |
|------|------|------|------|-------------|
| **zip**（现状） | 全量迁移 | 完整、二进制安全 | 不可读、不能局部编辑 | 是 |
| **md**（新增） | 阅读、评审、git 版本控制 | 人类友好、可 diff | 表格解析复杂、易格式错乱 | ❌ **只导出，不导入** |
| **excel**（新增） | 客户/业务方编辑 | 结构化、大家会用 | 需 poi 依赖 | ✅ **主推导入格式** |

**结论**：
- **md** = 只读文档（供人看）
- **excel** = 可编辑的主配置格式（供人改，可反导入）
- **zip** = 全量二进制迁移（保留现状，用于系统间搬迁）

## 三、excel 文件结构（导入的主格式）

### 3.1 单个"接口配置" excel 文件

一个接口 = 一个 `.xlsx`（用户要求"不同的接口导出的文件是单独的文件"）。

命名规范：`{接口类型}_{接口名}_{导出时间}.xlsx`，示例：`QUERY_账户查询_20260721.xlsx`

Sheet 结构：

| Sheet 名 | 内容 | 示例列 |
|----------|------|--------|
| **元数据** | 接口基础信息 | 接口ID / 接口名 / 类型 / 数据源 / 状态 / 描述 |
| **表配置** | 主表+关联表 | 表名 / 别名 / JOIN类型 / JOIN条件 |
| **字段列表** | 输出/输入字段 | 字段名 / 来源表.列 / 中文名 / 数据类型 / 是否必填 |
| **条件配置** | WHERE / 过滤 | 字段 / 操作符 / 值来源(REQUEST/CONST/CALC) / 值 |
| **数据来源**（插入/修改用） | 每列数据从哪来 | 目标列 / 来源类型 / 请求字段 / 常量值 / 计算表达式 |
| **加工规则**（可选） | 字段加工 | 字段 / 策略 / 参数 |
| **缓存配置**（可选） | 缓存开关+TTL | cache_enabled / cache_ttl_seconds / cache_key_template |

**导入约定**：
- 表头固定，用户不能改
- 第一列固定放"字段编码"（英文，导入用），第二列放"字段名称"（中文，仅展示）
- 隐藏 Sheet `_meta` 存 schemaVersion，导入时校验兼容性

### 3.2 单个"转换模板" excel 文件

命名：`TEMPLATE_{模板名}_{srcFormat}to{targetFormat}_{时间}.xlsx`

Sheet：
| Sheet | 内容 |
|-------|------|
| 元数据 | 模板名 / 源格式 / 目标格式 / 功能号 |
| 字段映射 | 源字段路径 / 目标字段路径 / 是否必填 |
| 字段加工 | 目标字段 / 加工策略 / 参数 |

## 四、md 文件结构（只读文档）

按 markdown 表格 + 代码块的形式生成，示例（接口配置）：

```markdown
# 接口：账户查询（QUERY_ACCOUNT）

- **接口ID**：123
- **类型**：QUERY（查询接口）
- **状态**：published
- **数据源**：mysql_business
- **描述**：按用户ID查询账户余额

## 主表 · 关联表
| 表名 | 别名 | JOIN | 条件 |
|------|------|------|------|
| account | a | (主表) | — |
| user | u | LEFT JOIN | a.user_id = u.id |

## 输出字段
| 字段名 | 来源 | 中文名 | 类型 |
| ... |

## WHERE 条件
| 字段 | 操作符 | 值来源 | 值 |
| a.user_id | = | REQUEST | userId |
```

**转换模板**类似结构（表格 + JSON 代码块贴映射规则）。

## 五、导入 · 交互与冲突策略

### 5.1 上传约束
- 一次可以选多个 excel（用户要求）
- 但**必须同类型**（要么全是"接口配置"，要么全是"转换模板"）—— 前端校验文件名前缀
- 若混选，弹窗提示"请分批导入不同类型"

### 5.2 冲突策略（继承现有 zip 导入）
| 策略 | 说明 |
|------|------|
| SKIP | 已存在则跳过，只导入新增 |
| OVERWRITE | 已存在则覆盖 |
| ASK | 仅检测冲突，返回冲突列表让用户挑（现有 UI 已支持） |

### 5.3 校验错误处理
- 逐 sheet / 逐行校验
- 校验失败**中止该文件**，其他文件继续
- 返回结果结构：
  ```json
  {
    "imported": ["接口A", "接口B"],
    "skipped": ["接口C（已存在）"],
    "failed": [
      {"file": "接口D.xlsx", "row": 12, "reason": "字段 xxx 数据类型不合法"}
    ]
  }
  ```

## 六、与「接口文档」菜单的打通

### 6.1 现状
- `/interface/doc` (InterfaceDocument.vue)：已经有"转换接口文档"和"可视化接口文档"两个 Tab，可以下载 md/HTML/zip
- `/interface/import-export` (InterfaceImportExport.vue)：只支持 zip

### 6.2 打通方案
两个菜单**合并为一个界面**（都在「辅助工具」大类下），三 Tab：

| Tab | 内容 |
|-----|------|
| **文档预览** | 列表 + 在线预览渲染 md（保留现状） |
| **导出** | 勾选接口 → 选择格式（md / excel / zip）→ 下载 |
| **导入** | 上传 excel（多选同类型）→ 冲突策略 → 结果反馈 |

顶部工具栏保留"全量导出 zip"作为快捷入口。

### 6.3 若不合并（备选）
保留两个菜单，但共用后端 API，`/interface/doc` 页面上加"编辑并导入"按钮，跳到 `/interface/import-export`。

**推荐**：合并为一个页面，更清晰。

## 七、后端改动清单

| 变更点 | 位置 | 说明 |
|--------|------|------|
| 新增 `POST /api/config/export/excel` | ConfigImportExportController | 参数：id列表、type（interface/template）；返回：zip 打包多 excel 或单 excel |

| 依赖 | pom.xml | 加 `poi-ooxml`（约 3MB，可接受） |

## 八、待用户确认的决策点

| # | 决策点 | 我的建议 |
|---|--------|---------|
| 1 | md 是否也支持"反导入" | **不支持**。md 表格解析很脆，逼用户改 excel 更靠谱 |
| 2 | excel 是每接口独立文件还是可以打包在一个 xlsx 里（用 sheet 分） | 每接口独立文件（用户明确要求） |
| 3 | 「接口文档」和「配置导入/导出」是否合并成一个页面 | **合并**。功能有大量重叠，两个菜单会让用户困惑 |
| 4 | 导入 excel 支持"部分字段更新" (patch) 还是"全量覆盖" | 全量覆盖，更简单也更安全 |
| 5 | 转换模板的字段映射用 excel 表格能表达吗？（源字段路径可能是嵌套 JSON） | 用 `.` 分隔路径（如 `user.contact.email`），可行 |
