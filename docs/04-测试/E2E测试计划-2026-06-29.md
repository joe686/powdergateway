# PowerGateway 全站 E2E 验证 + 自动修复测试计划

日期：2026-06-29
执行模式：自主闭环，不向用户提问，一次性跑完后输出汇总报告
范围：28 个交付单元 + 1 个综合首页

## 目标与不在范围

**目标**

- 验证 28 个交付单元的核心用户路径在浏览器中端到端可用（输入 → 处理 → 输出符合配置）
- 捕获并修复期间触发的页面报错、接口异常、控制台错误、路由跳转异常
- 输出可复核的汇总报告（覆盖率 / 已修 / 未修原因）

**不在范围**

- 性能压测、稳定性长跑
- 信创/国产化兼容性验证
- 后续优化项（OPT-x / FUT-x）
- 三库合并改造（用户已确认现阶段不做）

## 执行顺序

1. 启动 backend + frontend + pg-testkit
2. P0 优先：登录 → 报文转换核心链 → 接口配置核心链（M2-1/2/3/4/5/6/7）+ 执行调用 → 首页
3. P1 跟进：模板/渠道/分发、缓存、分片、审计、日志、性能、权限、配置、向导
4. P2 兜底：调试工具
5. 汇总报告

## 修复策略

- **明确 BUG**（页面崩溃、接口 500、控制台 ERROR、跳转异常）：当场定位代码并修复，写注释说明根因；不重启服务而是依赖 devtools hot reload / Spring DevTools 重载，必要时手动重启
- **疑似但无法复现**：在报告中记录"待复现"，不强行修
- **超出 BUG 范畴的改进点**（UX、性能、扩展性）：仅记录，不修
- **变更记录**：所有修复都登记 `docs/03-开发/变更记录.md` 的新增 `CHG-009`（或更新现有 CHG）

## 浏览器测试约定

- 每个用例：导航 → snapshot → 操作 → snapshot → 校验文本/dom → 收集控制台报错
- 用例间不强制隔离，依赖 Sa-Token session
- 数据用例尽量使用预置 admin 用户 + 临时创建 + 末尾清理（必要时跳过清理）

## 失败兜底

- 单个用例失败：记录到报告，继续下一个
- 整模块阻塞（如服务无法启动）：尝试一次重启，仍失败则跳过该模块的剩余用例
- 不向用户中断提问
