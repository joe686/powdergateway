## 一、现状 vs 用户需求的 gap

现有 `pg-testkit`（见 `测试工具计划.md`）设计得非常完整，但目标读者是 **AI（TRAE）** —— 通过 `/test/*` API 让 AI 驱动测试。

用户的诉求是：**"我可以按照我的要求测试的一个工具"**，即：

| 现状定位（AI 驱动） | 用户诉求（用户驱动） |
|--------------------|----------------------|
| Mock Server 通过 HTTP API 配置 | Mock Server 通过 GUI 配置 |
| 测试场景写 JSON 让 AI 跑 | 测试场景在页面上点点鼠标就跑 |
| 样例数据靠 `TestDataFactory` 编程生成 | 样例业务库自带完整的表和数据，一键初始化 |
| 断言在测试类里写 assert | 断言可视化对比：预期 vs 实际，红/绿高亮 |

**结论**：pg-testkit 的骨架不动，在其上加一层"用户友好"的表现层。

## 二、需要新增的两个能力

### 能力 A：样例业务数据库（供"可视化接口开发"练手）

一个自带完整表结构 + 初始数据的样例库，模拟真实客户业务系统。

### 能力 B：可视化 Mock 后端（供"接口转换"练手）

一个自带 GUI 的 Mock 服务，模拟外部系统接收 PG 转发的报文并返回预设响应。

## 三、能力 A · 样例业务库

### 3.1 数据库定位

- **数据库名**：`pg_demo_biz`（业务库）+ `pg_demo_biz_shard_01/02/03`（分库分表 demo 用）
- **DB 类型**：MySQL（首选，最常见），可选 H2（无 MySQL 时降级）
- **初始化**：pg-testkit 启动时自动建库建表灌数据，重复启动不重复灌

### 3.2 表设计（覆盖"可视化接口"所有能力）

设计一个"小型电商 + 银行"混合业务域，一次覆盖：单表、多表 JOIN、软删、字典、日期、金额、大文本、分库分表、缓存友好型表。

| # | 表名 | 用途 | 覆盖的接口能力 |
|---|------|------|---------------|
| 1 | `demo_user` | 用户表（1000 条） | 查询单表、缓存 |
| 2 | `demo_account` | 账户表（3000 条，用户1:N账户） | 查询多表 JOIN |
| 3 | `demo_txn` | 交易流水（10 万条，每账户~30 笔） | 大数据量、分页、分库分表 |
| 4 | `demo_product` | 商品表（500 条） | 字段加工、字典转换 |
| 5 | `demo_order` | 订单表（5000 条） | 插入接口、多表事务 |
| 6 | `demo_order_item` | 订单明细（1万条，1:N） | 多表插入事务 |
| 7 | `demo_address` | 地址表 | 修改接口 |
| 8 | `demo_dict` | 字典表（币种、状态、类型） | 字典 JOIN 转换 |
| 9 | `demo_config` | 大文本 JSON 表 | 复杂字段处理 |
| 10 | `demo_log` | 日志表（1 万条） | 删除接口、按日期删 |

**分库分表 demo**：`demo_txn` 按 `user_id % 3` 拆到 3 个物理库 `pg_demo_biz_shard_01/02/03`。

### 3.3 表结构示例（`demo_user`）

```sql
CREATE TABLE demo_user (
  id BIGINT AUTO_INCREMENT PRIMARY KEY,
  user_no VARCHAR(32) UNIQUE COMMENT '用户号',
  name VARCHAR(64) COMMENT '姓名',
  gender TINYINT COMMENT '性别 1男2女',   -- 字典演示
  birth_date DATE COMMENT '生日',
  phone VARCHAR(20),
  email VARCHAR(128),
  level TINYINT DEFAULT 1 COMMENT '等级 1-5', -- 加工演示
  balance DECIMAL(18,2) DEFAULT 0 COMMENT '总余额', -- 金额演示
  status TINYINT DEFAULT 1 COMMENT '状态', -- 软删演示
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  deleted TINYINT DEFAULT 0
);
```

（其他表结构在实际落地时补齐，这里省略。）

### 3.4 初始数据生成器

用 Java 代码 + `Faker` 库（Java版）生成语义合理的数据：
- 姓名：中文
- 手机号：13/15/17/18 开头 11 位
- 金额：正态分布，均值 5000
- 交易时间：过去 90 天内均匀分布

### 3.5 一键初始化 API

pg-testkit 暴露：
- `POST /testkit/demo-db/init` → 建库建表灌数据（幂等，可重跑）
- `POST /testkit/demo-db/reset` → 清空并重灌
- `GET /testkit/demo-db/stats` → 各表数据量统计

对应 pg-testkit 前端加一个页面「样例数据库管理」（简单卡片式）。

### 3.6 预配置的样例接口

自动往 PG 配置库里灌 10 个"样例接口配置"，用户登录 PG 就能看到，直接调用体验：

| 接口名 | 类型 | 说明 |
|--------|------|------|
| `DEMO_USER_QUERY` | QUERY | 按 user_no 查用户 |
| `DEMO_USER_LIST` | QUERY | 分页查用户列表 |
| `DEMO_ACCOUNT_JOIN` | QUERY | user + account 多表 JOIN |
| `DEMO_USER_INSERT` | INSERT | 新增用户（含数据来源演示） |
| `DEMO_USER_UPDATE` | UPDATE | 按 id 改用户等级+余额 |
| `DEMO_ORDER_CREATE` | INSERT | 多表插入订单+明细（事务演示） |
| `DEMO_ORDER_DELETE` | DELETE | 删订单（含明细级联） |
| `DEMO_TXN_QUERY_SHARDED` | QUERY | 分库分表查交易流水 |
| `DEMO_USER_CACHED` | QUERY | 带缓存的用户查询 |
| `DEMO_LOG_BATCH_DELETE` | DELETE | 批量删除（带保护开关演示） |

## 四、能力 B · 可视化 Mock 后端

### 4.1 定位

在 pg-testkit 里跑一个"可视化 Mock 服务"（默认端口 9999），提供 GUI 让用户：
- 定义规则："收到 `POST /cbs/query`，如果 body 包含 `userId=123`，返回 xxx"
- 查看历史请求：PG 发过来的每个请求，Header、Body、时间戳，可回放
- 编排响应：延时、状态码、Content-Type、Body 模板

### 4.2 界面草图

```
┌─────────────────────────────────────────────────────────────┐
│ Mock 后端服务 · 端口 9999          [启动/停止]  [清空历史]    │
├──────────────────┬──────────────────────────────────────────┤
│ 响应规则         │ 匹配                                       │
│                  │  Method: [POST ▼]                        │
│  [+ 新增规则]    │  Path: [/cbs/query]                      │
│                  │  Body 包含: [userId=123]                 │
│  ✓ CBS查询-成功  │                                          │
│  ✓ CBS查询-失败  │ 响应                                       │
│  ✓ 慢响应5秒     │  状态码: [200]                            │
│                  │  延迟: [0]ms                             │
│                  │  Content-Type: [application/json ▼]     │
│                  │  Body:                                   │
│                  │  ┌──────────────────────────────────┐    │
│                  │  │ { "code": 0, "balance": 12345 } │    │
│                  │  └──────────────────────────────────┘    │
├──────────────────┴──────────────────────────────────────────┤
│ 请求历史（最近 100 条）                                       │
│  时间               Method  Path         状态  耗时  详情    │
│  10:23:15           POST    /cbs/query   200   3ms  [查看]  │
│  10:23:20           POST    /cbs/query   500   3ms  [查看]  │
└─────────────────────────────────────────────────────────────┘
```

### 4.3 后端实现

pg-testkit 里跑一个 Servlet：
- `MockDispatcherServlet` 接收所有请求，按规则匹配
- 规则存内存（+ 每次修改持久化到 `pg-testkit/data/mock-rules.json`）
- 请求历史存 SQLite（`pg-testkit/data/mock-history.db`），避免占内存

匹配算法：
- Method 精确匹配
- Path 支持通配符 `/cbs/**`
- Body 支持"包含字符串"、"JSONPath 表达式"两种
- Header 同 Body

### 4.4 前端实现

pg-testkit 独立前端（Vite），复用 PowerGateway 前端的 Element Plus 主题，风格一致。

## 五、集成到 PowerGateway 主 UI 的入口

在 PG 前端「辅助工具」菜单下加一项：

- **样例数据 & Mock 后端**（跳转到 pg-testkit 前端）

链接前提：pg-testkit 已在 8081 端口跑起来（用户手动启动）。

或者：pg-testkit 前端页面 iframe 嵌到 PG 主 UI（更无缝但复杂）。

**推荐**：新页面跳转，简单可靠。

## 六、pg-testkit 打包与启动

### 6.1 独立启动脚本

在 pg-testkit 根目录提供：
- `start-testkit.bat`（Win）
- `start-testkit.sh`（Linux）

内容：
```bash
java -jar pg-testkit.jar --spring.profiles.active=demo
```

G 组打包方案里，`--with-testkit` 参数可以把 pg-testkit 一并打包，客户环境可选装。

## 七、待用户确认的决策点

| # | 决策点 | 我的建议 |
|---|--------|---------|
| 1 | 样例业务库用 MySQL 还是内嵌 H2？ | **两种都支持**。MySQL 首选（真实），H2 兜底（无 MySQL 也能跑） |
| 2 | pg-testkit 前端要不要独立 UI，还是复用 PG 前端 | **独立 UI**。数据隔离清晰，独立发布 |
| 3 | Mock 规则是否需要"录制回放"（拦截真实调用生成规则）？ | **v2 再做**。v1 手动配置规则足够 |
| 4 | 样例接口是否要覆盖所有接口类型（QUERY/INSERT/UPDATE/DELETE/分库分表/缓存）？ | **是**，用户就是想体验全功能 |
| 5 | 交易流水表 `demo_txn` 10 万条数据是否太多？ | 可以，MySQL 完全 hold 得住；初始化耗时 20s 左右可接受 |
| 6 | Mock 后端启用端口 9999 是否可配置 | 可配置，`application.yml` 里改 |

## 九、验收标准

用户能独立完成以下一整套流程，无需 AI 辅助：

1. `start-testkit.bat` 启动测试工具
2. 打开 http://localhost:8081，点"初始化样例数据库" → 看到"1000 用户、3000 账户、10万交易"提示
3. 打开 PowerGateway (http://localhost:5173)，登录 → 「可视化接口开发 → 接口管理」里看到 10 个 `DEMO_*` 样例接口
4. 直接测试调用 `DEMO_USER_QUERY`，输入 `user_no=U000001`，看到返回结果
5. 在 pg-testkit 里"启动 Mock 后端"，加一条规则 → 在 PowerGateway 里配置端口路由指向 `http://localhost:9999/cbs/query`
6. 发起转换调用，在 pg-testkit 请求历史里看到 PG 发过来的报文
