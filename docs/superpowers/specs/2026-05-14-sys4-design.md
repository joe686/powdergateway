## 一、背景与价值

### 问题

系统存在多个运行时参数（告警阈值、日志保留天数、缓存 TTL 等），当前分散在各服务中通过 `sysConfigMapper.selectById(key)` 每次查 DB 读取，没有统一的管理入口和 UI，修改参数需直接操作数据库。

### SYS-4 带来的三个提升

1. **参数可运营**：运维人员通过页面修改参数，无需接触数据库，无需重启应用。
2. **热更新生效**：`SysConfigService` 维护内存 Map，修改后立即刷新，各服务下次读取即得新值；`ConfigChangedEvent` 触发 Redis 模板缓存 evict，让 TTL 变更立即生效。
3. **配置可观测**：一个页面展示全部参数当前生效值，运维排查时不用猜。

### 后续升级方向（暂不实现）

方案二：将 `SysConfigService` 的内存 Map 升级为 Redis 缓存，支持多实例部署时配置同步。

## 二、数据库变更

### `sys_config` 表新增两列

```sql
ALTER TABLE sys_config
  ADD COLUMN value_type  VARCHAR(32)  DEFAULT 'string' COMMENT '控件类型：number/boolean/string',
  ADD COLUMN group_name  VARCHAR(64)  DEFAULT '其他'   COMMENT '前端分组名称';
```

## 三、后端设计

### 3.1 `SysConfig` 实体类更新

新增 `valueType` 和 `groupName` 两个字段，对应 `sys_config` 表新增列。

### 3.2 `SysConfigService`

系统内读取 `sys_config` 的**唯一入口**，替代各处直接注入 `SysConfigMapper`。

```
职责：
  - @PostConstruct init()：启动时从 DB 加载全部 KV 到 ConcurrentHashMap
  - getString(key, defaultVal)
  - getInt(key, defaultVal)
  - getBoolean(key, defaultVal)
  - getAll()：返回 DB 全量列表（含 description/valueType/groupName），供前端展示
  - batchUpdate(Map<String,String> updates)：
      1. 写 DB（有则 updateById，无则 insert）
      2. 更新内存 cache（立即生效）
      3. 发布 ConfigChangedEvent（携带 changedEntries）
```

### 3.3 `ConfigChangedEvent`

```java
public class ConfigChangedEvent extends ApplicationEvent {
    private final Map<String, String> changedEntries;
}
```

### 3.4 ApplicationEvent 监听者

`ConvertService`（或 `TemplateController` 所在 Service）监听 `ConfigChangedEvent`：
- 若 `changedEntries` 含 `cache.template.ttl`，执行 `@CacheEvict(value="template", allEntries=true)`，清除 Redis 中全部模板缓存

### 3.5 改造现有服务

将以下位置的 `sysConfigMapper.selectById(key)` 替换为 `sysConfigService.getXxx(key, default)`：

| 文件 | 改造前 | 改造后 |
|---|---|---|
| `PerfAlertJob` | `sysConfigMapper.selectById("alert_fail_rate")` | `sysConfigService.getInt("alert_fail_rate", 5)` |
| `PerfAlertJob` | `sysConfigMapper.selectById("alert_response_ms")` | `sysConfigService.getInt("alert_response_ms", 1000)` |
| `AuditLogCleanupJob` | `sysConfigMapper.selectById("audit.log.retention.days")` | `sysConfigService.getInt("audit.log.retention.days", 365)` |
| `SysLogArchiveJob` | `sysConfigMapper.selectById("sys.log.retention.days")` | `sysConfigService.getInt("sys.log.retention.days", 30)` |
| `AuthService` | `sysConfigMapper.selectById("log_menu_enabled")` | `sysConfigService.getBoolean("log_menu_enabled", true)` |
| `PerfStatService.upsertConfig` | 直接操作 `SysConfigMapper` | 改为调用 `sysConfigService.batchUpdate()` |

### 3.6 `SysConfigController`

```
GET  /api/config/all    返回全部 KV 列表（含 groupName/valueType/description/updateTime）
PUT  /api/config        批量更新，Body: Map<String, String>
                        @SaCheckRole("admin") 保护，非管理员返回 403
```

## 四、前端设计

### 4.1 数据驱动渲染逻辑

```
后端返回: [{ configKey, configValue, description, valueType, groupName, updateTime }, ...]

前端处理:
  1. 按 groupName 分组 → 渲染多个 el-card/el-divider
  2. 每项按 valueType 渲染控件:
       number  → el-input-number
       boolean → el-switch
       string  → el-input
       (未来) select → el-select / textarea → el-input type="textarea"
  3. description 展示为灰色小字
  4. 非 admin 角色：保存按钮禁用
```

### 4.2 文件

| 文件 | 说明 |
|---|---|
| `frontend/src/views/system/SystemConfig.vue` | 新建，替换 PlaceholderView |
| `frontend/src/api/sysConfig.js` | `getAllConfig()` / `updateConfig(data)` |
| `frontend/src/router/index.js` | 替换 `system/config` 路由指向 |

### 4.3 用户体验

- 进入页面自动加载当前值
- 「保存」提交全部当前值，成功提示"配置已保存，立即生效"
- 非 admin 角色保存按钮置灰（可查看，不可修改）

## 六、验收标准

1. 修改告警阈值后，下次 `PerfAlertJob` 执行使用新值（无需重启）
2. 修改 `cache.template.ttl` 后，Redis 模板缓存被清除，新缓存写入使用新 TTL
3. 关闭 `log_menu_enabled` 后，下次 `/api/auth/menu` 返回中不含日志菜单项
4. 前端按分组展示全部 KV，数字/开关控件正确渲染
5. 非 admin 角色进入页面，「保存」按钮禁用
6. `mvn test` 全量测试通过，无回归
