# PowerGateway 需求最小化拆分与实现方案

> 基于《可视化接口程序产品需求说明书》拆分，每个单元可独立开发交付。
> 实现方案优先复用既定技术栈（Spring Boot / MyBatis-Plus / Sa-Token / Vue3 / Element Plus）内置能力。

---

## 总览

| 分组 | 单元数 | 说明 |
|------|--------|------|
| P0 基础脚手架 | 4 | 所有功能的前置依赖，一次性完成 |
| M1 报文格式转换 | 7 | 模块一全部功能（含端口分发路由） |

## P0 基础脚手架

所有后续功能的基础，需最先完成。

---

### P0-1 后端项目初始化

- **范围**：可运行的 Spring Boot 骨架，含基础依赖和统一响应结构
- **不含**：任何业务逻辑
- **实现方案**：
  1. 使用 Spring Initializr 生成项目（Group: `com.powergateway`，Artifact: `backend`）
  2. `pom.xml` 引入：`mybatis-plus-boot-starter`、`sa-token-spring-boot-starter`、`springdoc-openapi-ui`、`spring-boot-starter-data-redis`、`caffeine`、`jackson-dataformat-xml`、`dom4j`、`opencsv`
  3. `application.yml` 配置：端口 8080、数据库连接（配置库）、Redis 连接
  4. 封装统一响应体 `Result<T>`（code / message / data）
  5. 全局异常处理 `@RestControllerAdvice`
- **前置依赖**：无
- **验收标准**：`mvn spring-boot:run` 启动后访问 `http://localhost:8080/swagger-ui.html` 页面正常

---

### P0-2 前端项目初始化

- **范围**：可运行的 Vue3 骨架，含路由、状态管理、HTTP 封装、布局框架
- **不含**：任何业务页面
- **实现方案**：
  1. `npm create vue@3` 生成项目（`frontend/`），选择 Vue Router + Pinia
  2. 安装：`element-plus`、`axios`、`vue-draggable-next`、`echarts`、`vue-echarts`
  3. `vite.config.js` 配置 API 代理（`/api` → `http://localhost:8080`）
  4. 封装 `src/api/request.js`：axios 实例，统一添加 token 请求头，统一处理 401/500
  5. 实现基础布局组件（侧边菜单 + 顶栏 + 内容区），菜单使用 `el-menu`
  6. `src/store/user.js`：Pinia store 存储登录用户信息和 token
- **前置依赖**：无
- **验收标准**：`npm run dev` 启动后浏览器访问 `http://localhost:5173` 显示登录页框架

### P0-3 配置库建表 SQL

- **范围**：配置库的全部核心表 DDL，及对应的 MyBatis-Plus 实体类
- **不含**：审计库（M2-9 单独建）
- **实现方案**：
  在 `backend/src/main/resources/db/` 创建 `init.sql`，包含以下 8 张表：

  ```sql
  -- 1. 用户表
  CREATE TABLE sys_user (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(64) NOT NULL UNIQUE,
    password VARCHAR(128) NOT NULL COMMENT 'BCrypt加密',
    role VARCHAR(32) DEFAULT 'user' COMMENT 'admin/user/readonly',
    status TINYINT DEFAULT 1,
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
  );

  -- 2. 转换模板表
  CREATE TABLE convert_template (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(128) NOT NULL,
    src_format VARCHAR(32) COMMENT 'JSON/XML/CSV/FORM',
    target_format VARCHAR(32),
    mapping_rule JSON COMMENT '字段映射规则列表',
    process_rule JSON COMMENT '字段加工规则列表',
    is_latest TINYINT DEFAULT 1,
    version INT DEFAULT 1,
    deleted TINYINT DEFAULT 0,
    creator VARCHAR(64),
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
  );

  -- 3. 渠道配置表
  CREATE TABLE channel_config (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    channel_code VARCHAR(64) NOT NULL UNIQUE COMMENT '渠道编码',
    channel_name VARCHAR(128),
    identify_field VARCHAR(128) COMMENT '识别字段名',
    template_id BIGINT COMMENT '关联convert_template.id',
    deleted TINYINT DEFAULT 0,
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP
  );

  -- 4. 数据库连接表
  CREATE TABLE db_connection (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(128) NOT NULL,
    db_type VARCHAR(32) COMMENT 'MySQL/Oracle/PostgreSQL',
    url VARCHAR(512) NOT NULL,
    username VARCHAR(128),
    password VARCHAR(256) COMMENT 'AES加密存储',
    env VARCHAR(32) DEFAULT 'dev' COMMENT 'dev/test/prod',
    pool_size INT DEFAULT 10,
    timeout INT DEFAULT 3000,
    deleted TINYINT DEFAULT 0,
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP
  );

  -- 5. 接口配置表
  CREATE TABLE interface_config (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(128) NOT NULL,
    path VARCHAR(256) COMMENT '发布后的访问路径',
    type VARCHAR(32) COMMENT 'SELECT/INSERT/UPDATE/DELETE',
    db_connection_id BIGINT,
    config_json JSON COMMENT '完整接口配置（表、字段、条件、加工规则等）',
    shard_config_id BIGINT COMMENT '关联分库分表配置',
    allow_batch_delete TINYINT DEFAULT 0,
    status VARCHAR(32) DEFAULT 'draft' COMMENT 'draft/published/disabled',
    log_enabled TINYINT DEFAULT 1,
    deleted TINYINT DEFAULT 0,
    creator VARCHAR(64),
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
  );

  -- 6. 分库分表配置表
  CREATE TABLE shard_config (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(128) NOT NULL,
    module_name VARCHAR(128) COMMENT '业务模块名',
    request_field VARCHAR(128) COMMENT '请求中的路由字段',
    shard_rule JSON COMMENT '库表映射规则',
    deleted TINYINT DEFAULT 0,
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP
  );

  -- 7. 常用字段公式表
  CREATE TABLE field_formula (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(128) NOT NULL UNIQUE,
    scene VARCHAR(128) COMMENT '所属业务场景',
    db_connection_id BIGINT,
    formula_json JSON COMMENT '公式配置（条件、运算、接口字段关联）',
    remark VARCHAR(512),
    deleted TINYINT DEFAULT 0,
    creator VARCHAR(64),
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP
  );

  -- 8. 系统配置表（KV）
  CREATE TABLE sys_config (
    config_key VARCHAR(128) PRIMARY KEY,
    config_value VARCHAR(1024),
    description VARCHAR(512),
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
  );

  -- 9. 端口分发路由表（M1-7）
  CREATE TABLE port_route (
    id                   BIGINT PRIMARY KEY AUTO_INCREMENT,
    channel_code         VARCHAR(64)  NOT NULL COMMENT '渠道编码，关联 channel_config.channel_code',
    port_address         VARCHAR(512) NOT NULL COMMENT '目标端口完整URL（含协议和路径）',
    port_method          VARCHAR(16)  DEFAULT 'POST'  COMMENT 'GET/POST/PUT/DELETE',
    timeout              INT          DEFAULT 3000    COMMENT '连接超时（ms）',
    retry_count          INT          DEFAULT 3       COMMENT '失败重试次数',
    request_template_id  BIGINT       COMMENT '请求方向转换模板ID（A→B），关联 convert_template.id',
    response_template_id BIGINT       COMMENT '应答方向转换模板ID（B→A），为空则透传',
    deleted              TINYINT      DEFAULT 0,
    create_time          DATETIME     DEFAULT CURRENT_TIMESTAMP
  );
  ```

  使用 MyBatis-Plus Generator 或手写实体类（`@TableName`、`@TableId`、`@TableLogic`）。

- **前置依赖**：P0-1
- **验收标准**：`init.sql` 执行无报错，9张表结构正确

### P0-4 用户登录鉴权

- **范围**：登录/登出接口 + Sa-Token 全局拦截 + 前端路由守卫
- **不含**：用户管理界面（SYS-3 处理）
- **实现方案**：
  - 后端：
    - `POST /api/auth/login`：校验用户名/密码（BCrypt），调用 `StpUtil.login(userId)` 返回 token
    - `POST /api/auth/logout`：调用 `StpUtil.logout()`
    - `SaInterceptor` 注册拦截所有 `/api/**`，排除 `/api/auth/**`
  - 前端：
    - `router.beforeEach` 检查 Pinia store 中是否有 token，无则跳转 `/login`
    - 初始化时预置一个管理员账号（`admin / Admin@123`）
- **前置依赖**：P0-1、P0-2、P0-3
- **验收标准**：登录成功后能访问任意页面，未登录跳转到登录页

## M1 模块一：报文格式转换

### M1-1 格式转换引擎

- **范围**：JSON / XML / CSV / FormData 四种格式的互相转换（后端工具类）+ 报文格式转换前端页面（含模板选择调用）
- **不含**：字段映射配置界面（M1-2）、字段加工规则配置界面（M1-3）
- **实现方案**：
  - 后端：
    - 创建 `utils/FormatConverter.java`，方法签名：
      ```java
      public String convert(String src, FormatType from, FormatType to)
      ```
    - 转换逻辑（各格式先统一解析为 `Map<String, Object>`，再序列化为目标格式）：
      - JSON：`ObjectMapper.readValue()` / `writeValueAsString()`
      - XML：`Dom4j Document.parse()` / `Element.asXML()`
      - CSV：`OpenCSV CSVReader` / `CSVWriter`
      - FormData：`URLDecoder.decode()` / `URLEncoder.encode()`
    - `FormatType` 枚举：`JSON, XML, CSV, FORM_DATA`
  - 前端（`views/convert/FormatConvert.vue`）：
    - 顶部"使用模板（可选）"下拉，列出所有已配置模板（显示 `名称（srcFmt → targetFmt）`），支持搜索和清除
    - 选中模板后：自动填充并锁定源格式/目标格式，右侧显示"含字段映射 + 字段加工"标签；转换时调用 M1-6 全流程接口 `POST /api/convert`（携带 `templateId`）
    - 未选模板时：手动选择源格式/目标格式，调用 `POST /api/format-convert/convert` 直接转换，并行调用 `/parse` 展示字段解析预览
    - 快速填充示例（JSON/XML/CSV/FormData）、复制结果、一键重置
- **前置依赖**：P0-1、P0-2（前端部分还依赖 M1-6 已完成后才能使用模板功能）
- **验收标准**：
  - 后端：单元测试覆盖 12 种转换组合（排除 FormData↔FormData），断言转换后字段值一致
  - 前端（无模板）：选格式、输入报文、点击转换，右侧显示转换结果和字段解析预览
  - 前端（有模板）：选中已有模板后格式自动填充锁定，点击转换调用全流程接口，结果正确返回

### M1-2 字段映射配置

- **范围**：可视化配置源字段到目标字段的映射关系，保存为模板的一部分
- **不含**：格式转换执行（M1-6）、字段加工（M1-3）
- **实现方案**：
  - 后端：
    - `convert_template.mapping_rule` JSON 字段存储映射列表：
      ```json
      [{"srcField": "user_id", "targetField": "userId", "fixedValue": null}]
      ```
      - `srcField` 支持点号路径访问嵌套字段（CHG-004），例如 XML `<root><head><FunctionId>` 对应 `"head.FunctionId"`；无点号时为平铺字段名，向后兼容
    - `POST /api/template/save`：保存模板含映射规则
    - `GET /api/template/{id}/preview`：传入测试报文，应用映射规则返回预览结果（调用 M1-1）
  - 前端（`views/convert/FieldMapping.vue`）：
    - 左树（源字段）+ 右树（目标字段），使用 `el-tree`
    - 拖拽源字段到目标字段完成映射：`vue-draggable-next`
    - 自动匹配按钮：前端 computed，字段名相同时自动填充映射
    - 固定值填入：点击目标字段旁边「固定值」按钮，弹出 `el-input` 输入
- **前置依赖**：P0-1、P0-2、P0-3、M1-1
- **验收标准**：配置映射后保存，重新打开模板映射关系正确还原；预览结果字段映射符合配置

---

### M1-3 字段加工规则引擎

- **范围**：6类加工（截位/补位/格式转换/大小写/去空格）+ 多规则叠加执行
- **不含**：前端配置界面（含在本单元）
- **实现方案**：
  - 后端（策略模式）：
    ```java
    interface FieldProcessStrategy {
        String process(String value, Map<String, String> params);
    }
    // 实现类：TrimProcessor, PadProcessor, SubstringProcessor,
    //         CaseProcessor, FormatConverter, TypeCastProcessor
    ```
    `FieldProcessor.java` 按规则列表顺序依次调用，`ProcessRule` 包含 `{type, params}`
  - 加工规则存入 `convert_template.process_rule` JSON
  - **复用**：M2 查询/插入/修改的字段加工直接注入同一个 `FieldProcessor` Bean
  - 前端（`views/convert/FieldProcess.vue`）：
    - 规则列表，每条规则：选择类型 → 填写参数 → 实时预览效果
    - `vue-draggable-next` 控制规则顺序（上移/下移）
- **前置依赖**：P0-1、P0-2
- **验收标准**：输入 `"  hello world  "`，配置「去空格 → 首字母大写 → 截取前5位」，输出 `"Hello"`

### M1-4 渠道模板管理

- **范围**：渠道定义、渠道与转换模板的关联、运行时按渠道字段自动选模板、**渠道级别报文头默认配置**（CHG-002）
- **不含**：端口分发（需求文档 3.1.5，列为后期迭代）；Header 预览/测试功能；入向请求 Header 校验
- **实现方案**：
  - 后端：
    - `GET /api/channel/list`、`POST /api/channel/save`、`DELETE /api/channel/{id}`
    - 渠道匹配逻辑 `ChannelRouter.match(String message, FormatType type)`：
      1. 解析报文提取 `identify_field` 字段值
      2. 查 `channel_config` 表（Redis 缓存）匹配 `channel_code`
      3. 返回关联的 `template_id`
    - `channel_config` 表新增 `header_config TEXT` 列（JSON），存储 `HeaderConfig`：
      `contentType`（出向 Content-Type）、`charset`（字符集）、`requestHeaders`（自定义出向 KV）、`responseHeaders`（返回 A 的响应头 KV）
    - `ChannelSaveRequest` 新增 `headerConfig` 字段，`ChannelConfigService.save()` 序列化存储
  - 前端（`views/convert/ChannelConfig.vue`）：
    - 渠道列表（`el-table`）：渠道编码、渠道名、识别字段、关联模板
    - 新增/编辑弹窗：`el-form`，模板字段使用 `el-select` 从已有模板中选择
    - 弹窗底部新增"报文头配置"折叠区（`el-collapse`）：Content-Type 选择器、Charset 选择器、请求头 KV 表格、响应头 KV 表格
- **前置依赖**：P0-1、P0-2、P0-3、M1-5（需先有模板可选）
- **验收标准**：
  1. 配置渠道后，调用 `/api/convert` 传含渠道标识字段的报文，自动匹配对应模板执行转换
  2. 渠道 Header 配置保存后刷新页面数据不丢失
  3. dispatch 转发时，B 系统收到的请求头与渠道配置一致（无端口路由覆盖时）

### M1-6 报文转换调用接口（API 端点）

- **范围**：对外暴露转换接口，串联格式转换 → 字段映射 → 字段加工全流程
- **不含**：端口分发（M1-7 处理）
- **实现方案**：
  - 后端：
    ```
    POST /api/convert
    Body: { "templateId": 1, "message": "...", "srcFormat": "JSON" }
    Response: { "code": 0, "data": { "result": "...", "targetFormat": "XML", "costMs": 12 } }
    ```
    执行链：`读模板配置（Redis缓存）→ M1-1格式转换 → M1-2映射应用（支持点号路径，CHG-004）→ M1-3加工 → 返回`
  - Redis 缓存模板：`@Cacheable(value = "template", key = "#templateId")`，TTL 10分钟
  - 可选渠道自动匹配：若不传 `templateId`，传 `channelField` 自动走 M1-4 匹配
- **前置依赖**：M1-1、M1-2、M1-3、M1-5
- **验收标准**：Postman 调用接口，传 JSON 报文，返回正确 XML；响应时间 ≤ 200ms（带缓存）

### M1-7 端口分发路由

> **实现状态（2026-03-25）**：后端 + 前端均已完成。前端页面路径 `/convert/port-route`，对应 `views/convert/PortRoute.vue`，API 封装在 `api/portRoute.js`。
> **CHG-002（2026-03-27）**：新增报文头适配配置（两级：渠道默认 + 端口路由覆盖）、字节级 charset 转码。

- **范围**：渠道与目标端口的绑定配置（CRUD）+ 双向转换（请求加工→转发→应答加工）+ 调用接口 `POST /api/dispatch` + **端口路由级别报文头覆盖配置**（CHG-002）
- **不含**：异步回调模式（同步请求-应答场景已覆盖）；Header 预览/测试；入向请求 Header 校验；Header 加密存储
- **核心设计**：一条路由记录描述完整双向链路：`channel_code`（渠道）+ `port_address`（目标端口）+ `request_template_id`（A→B 请求转换模板）+ `response_template_id`（B→A 应答转换模板）
- **实现方案**：
  - `port_route` 表（含 CHG-002 新增列）：
    ```sql
    CREATE TABLE port_route (
      id                   BIGINT PRIMARY KEY AUTO_INCREMENT,
      channel_code         VARCHAR(64)  NOT NULL COMMENT '渠道编码，关联 channel_config.channel_code',
      port_address         VARCHAR(512) NOT NULL COMMENT '目标端口完整URL（含协议和路径）',
      port_method          VARCHAR(16)  DEFAULT 'POST'  COMMENT 'GET/POST/PUT/DELETE',
      timeout              INT          DEFAULT 3000    COMMENT '连接超时（ms）',
      retry_count          INT          DEFAULT 3       COMMENT '失败重试次数',
      request_template_id  BIGINT       COMMENT '请求方向转换模板ID（A→B），关联 convert_template.id',
      response_template_id BIGINT       COMMENT '应答方向转换模板ID（B→A），关联 convert_template.id，为空则透传',
      header_config        TEXT         COMMENT '端口路由级别报文头配置（JSON），覆盖渠道默认值（CHG-002）',
      deleted              TINYINT      DEFAULT 0,
      create_time          DATETIME     DEFAULT CURRENT_TIMESTAMP
    );
    ```
  - 后端接口：
    - `GET /api/port-route/list`：分页列表（支持渠道编码搜索）

      ```
      Body: { "message": "...", "srcFormat": "JSON" }

      执行链（CHG-002 更新）：
      1. ChannelRouter.match → 识别 channel_code
      2. 查 port_route（Redis缓存）+ 查 channel_config → 两级 HeaderConfig 合并
      3. [请求方向] 读 request_template → 格式转换+映射+加工（复用 M1-6 链路）

      Response: { "code": 200, "data": { "response": "...", "channelCode": "XX", "costMs": 30 } }
      ```
  - 新增 `HeaderConfigMerger` 工具类：两级合并，port_route 字段优先于 channel_config
  - 新增 `CharsetConverter` 工具类：字节级编码转换（UTF-8/GBK/ISO-8859-1）
  - Redis 缓存端口路由：`PortRoute` 对象含 `headerConfig`，无需额外改动
  - 转发实现：注入 `RestTemplate`（连接池复用），超时读自 `port_route.timeout`，失败循环重试 `retry_count` 次
  - 前端（`views/convert/PortRoute.vue`）：
    - 列表：渠道编码、端口地址、请求方式、请求模板、应答模板、操作列（编辑/删除/测试连通）
    - 新增/编辑弹窗：`el-form`
      - 渠道编码：`el-select`（从已有渠道配置选）
      - 请求转换模板：`el-select`（从已有转换模板选，必填）
      - 应答转换模板：`el-select`（从已有转换模板选，可不填=透传）
      - "报文头配置"折叠区：Content-Type 选择、Charset 选择、自定义 KV 表格（提示：未填则继承渠道默认值）
    - 删除使用 `el-popconfirm` 二次确认
- **前置依赖**：M1-4、M1-5、M1-6
- **验收标准**：
  1. 配置路由后调用 `/api/dispatch`，请求报文经请求模板转换后转发 B 系统
  2. B 系统应答经应答模板转换后返回给调用方
  3. 不配置应答模板时，B 系统原始应答直接透传

### M2-9 SQL 审计日志

- **范围**：INSERT/UPDATE/DELETE 操作异步写入独立审计库

### M2-10 缓存查询管理（双层缓存）

- **范围**：为 SELECT 接口配置 Caffeine + Redis 双层缓存，含命中统计
- **不含**：普通查询接口执行（M2-7 已实现）
- **实现方案**：
  - 数据模型：在 `interface_config` 新增字段 `cache_enabled`、`cache_ttl_seconds`、`cache_key_template`（支持 `{参数名}` 占位符）
  - 执行时双层缓存逻辑（`CacheManager.java`）：
    ```
    1. 生成 cacheKey = 模板替换请求参数
    2. 查 Caffeine（CaffeineCache）→ 命中直接返回
    3. 未命中查 Redis → 命中则写回 Caffeine，返回
    4. 未命中则加 Redis 分布式锁（SET NX PX 3000）→ 查 DB → 写 Redis + Caffeine → 释放锁
    ```
  - 命中统计：每次命中 Redis 时执行 `INCR cache_hit:{interfaceId}`；查 DB 时 `INCR cache_miss:{interfaceId}`
  - 前端（`views/cache/CacheList.vue`）：
    - 接口列表，显示：接口名、缓存 key 模板、TTL、命中次数、未命中次数
    - 操作：清除缓存（`@CacheEvict`）、立即刷新（清除后预热查询）
- **前置依赖**：M2-3、M2-7
- **验收标准**：第一次查询走 DB（缓存未命中），第二次查询命中 Caffeine，响应时间明显更短；命中统计数字正确

    - 告警配置弹窗：可修改失败率阈值和响应时间阈值
- **前置依赖**：P0-2、M2-7
- **验收标准**：执行10次查询后，统计页图表有数据；修改告警阈值后下次检查生效

### SYS-3 用户权限管理

- **范围**：用户增删改查 + 角色分配（admin/user/readonly）+ 菜单权限控制

- **前置依赖**：P0-3、P0-4
- **验收标准**：readonly 角色用户无法看到删除菜单和分库分表配置菜单

### SYS-4 系统配置

- **范围**：全局配置管理（缓存TTL、日志留存天数、告警阈值等）
- **实现方案**：

  - 前端（`views/system/SystemConfig.vue`）：

- **前置依赖**：P0-3

### SYS-5 接口配置 9 步导航

- **范围**：将 M2-3～M2-7 已有表单包装为步骤向导，不新增后端接口
- **实现方案**：
  - 纯前端实现（`views/interface/InterfaceWizard.vue`）：
    - `el-steps` 组件展示9个步骤，当前步骤高亮
    - 每步骤内容为已有子组件（`<QueryConfig>`/`<InsertConfig>` 等），按接口类型动态显示
    - 步骤数据统一存入 `wizardStore`（Pinia），步骤切换时持久化到 `localStorage`
    - 步骤9「保存与发布」调用 `POST /api/interface/save` + `POST /api/interface/{id}/publish`
    - 每步骤右上角悬浮「?」提示气泡（`el-tooltip`）
  - 9步映射：
    1. 选接口类型 → 简单 `el-radio-group`
    2. 选数据库连接 → `el-select` 从 M2-1 接口获取
    3. 选表结构 → M2-2 的树形组件
    4. 分库分表（可选）→ `el-select` + 跳过按钮
    5. 字段加工（可选）→ M1-3 的规则列表组件
    6. 条件配置 → M2-3/5/6 的条件配置组件
    7. 日志开关 → `el-switch`
    8. 预览测试 → 调用 `/api/interface/{id}/preview`
    9. 保存发布 → 填写名称/路径，调用发布接口
- **前置依赖**：M2-3、M2-4、M2-5、M2-6、M2-7（前端复用）
- **验收标准**：按步骤完成配置后发布，接口可正常调用；中途关闭再打开，进度从 localStorage 恢复

  - 公式版本历史（当前只软删）
