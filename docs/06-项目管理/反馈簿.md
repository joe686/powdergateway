# 反馈簿

> **单一入口**：所有需求反馈 / 问题反馈 / 视觉调整 / 新想法，都追加到本文件的[「活跃反馈区」](#三活跃反馈区)。
> **单一出口**：处理进度与结果直接回写本文件的**状态列**；重大范围变更同步归档到 [变更记录 CHG-XXX](../03-开发/变更记录.md)；发新版本时归档到 [路线图 · 已发布](./路线图.md#已发布)。

---

## 一、使用说明

### 你（用户）怎么反馈

在[「活跃反馈区」](#三活跃反馈区)的**顶部**追加一条，最简格式：

```markdown
### FB-XXX <一句话现象>
- 时间：YYYY-MM-DD
- 反馈：<你的原话，越具体越好，可以粘图>
```

`XXX` 是编号，从当前活跃反馈区最新那条 +1 递增。**你不用管分类、优先级、目标版本，我来补**。

### 我（Claude）接手后自动做的

1. **补齐条目字段**：分类（Bug / 新需求 / 视觉调整）、优先级（P0~P2）、目标版本、状态
2. **规划**：
   - Bug 且小 → 直接修
   - Bug 且大 或 新功能 → 在 `docs/03-开发/任务计划/YYYY-MM-DD-FB-XXX.md` 写详细任务分解（TDD）
   - 新需求 → 同步落一份 CR 条目到 [待办与缺陷池 § 变更申请](./待办与缺陷池.md#三变更申请模板)
3. **实施**：按 TDD 走 Red → Green → Refactor（后端加测试；前端跑 build + 相关 vitest）
4. **归档**：范围变更写 CHG-XXX 到 [变更记录.md](../03-开发/变更记录.md)；非范围变更不写 CHG
5. **回写状态**：把本反馈条目的状态列改到最新状态，加上 CHG 引用或 commit 短 SHA
6. **发版**：累计到打新 tag 时，同步 [路线图.md](./路线图.md) 已发布表

### 你在哪看结果

| 想看什么 | 去哪 |
|---|---|
| 我反馈的这条现在什么状态？ | 本文件[「活跃反馈区」](#三活跃反馈区)对应 FB-XXX 的状态列 |
| 具体做了什么改动？ | 本文件里 FB-XXX 引用的 CHG-XXX 或 commit SHA |
| 涉及哪些设计变更？ | [变更记录.md](../03-开发/变更记录.md) 对应 CHG |
| 修完在哪个版本发布？ | [路线图.md](./路线图.md) 已发布表 |
| 详细任务分解（大改） | `docs/03-开发/任务计划/` 对应 FB-XXX 文件 |

## 二、状态图例

| 图标 | 状态 | 含义 |
|------|------|------|
| 📥 | 待处理 | 你刚提交，我还没受理 |
| 🔍 | 已受理规划中 | 我在写方案/任务分解，未开始编码 |
| 🔧 | 开发中 | 代码在改 |
| 🧪 | 测试中 | 代码改完，跑测试或等你验证 |
| ✅ | 已交付 | 已合并 master + CHG-XXX 归档 + 版本已发（或待打 tag） |
| 🚫 | 已拒绝 | 明确不做（拒绝原因在条目里写清楚） |
| 💤 | 已延后 | 归入某个未来版本，见目标版本字段 |

## 五、维护约定

- **一 FB 一次响应**：一条反馈从接手到闭环内**不换编号**；即使拆分为多个 CHG，都关联到同一个 FB-XXX
- **拒绝要写清楚原因**：`🚫 已拒绝` 状态必须在条目下方补一段 `**拒绝原因**：xxx`
- **延后要指定版本**：`💤 已延后` 必须在状态列写清楚 `→ vX.Y.Z`
- **每次发新版本前**：把状态在 ✅ / 🚫 / 💤 的活跃反馈整理归档到「历史归档」的新批次段落
- **合并已完成任务**：若 FB-XXX 关联的任务计划文件也已完成，可选择性删除 `docs/03-开发/任务计划/YYYY-MM-DD-FB-XXX.md`（保留 CHG 记录即可追溯）
